# <a name="optimizing-windows-10-version-1803-for-a-virtual-desktop-infrastructure-vdi-role"></a>最佳化 Windows 10 版本 1803、 虛擬桌面基礎結構 (VDI) 角色

這篇文章可協助您選擇的 Windows 10 1803年版 （組建 17134） 應該導致在虛擬桌面基礎結構 (VDI) 環境中達到最佳的效能設定。 本指南中的所有設定都都*才會被視為建議*而且沒有方法的需求。

VDI 環境中將 Windows 10 的效能最佳化的索引鍵的方式是減少應用程式圖形重繪，沒有任何主要的好處，VDI 環境中，背景活動，並通常以最低限度減少執行中處理序。 次要的目標是要減少以最低限度的基底映像中的磁碟空間使用量。 Vdi 實作、 最小可能的基底或 「 黃金 」 的映像大小，都可以稍微降低 hypervisor，以及小型桌惎蝭萺傳遞給取用者所需的整體網路作業減少記憶體使用量。

> [!NOTE]  
> 這裡所建議的設定可以套用至 Windows 10 版本 1803，包括在實體或虛擬裝置上的其他安裝。 本主題中的沒有任何建議，應該會影響 Windows 10 1803年版的支援能力。

> [!TIP]  
> 本主題中，實作討論的最佳化的指令碼，以及您可以使用匯入 GPO 匯出檔案**LGPO.exe**-將會位於[TheVDIGuys](https://github.com//TheVDIGuys) GitHub 上。

## <a name="vdi-optimization-principles"></a>VDI 最佳化原則

VDI 環境提供完整的桌面工作階段，包括應用程式，以透過網路的電腦使用者。 VDI 環境通常會使用基底作業系統映像，就會變成後續將呈現給使用者的工作的桌面的基礎。 有多種變化 VDI 的實作，例如 「 持續性 」，「 非持續性 」，和 「 桌面工作階段 」。 永續性的型別會保留到下一個工作階段從 VDI 桌面作業系統的變更。 非持續性的型別不會保留變更 VDI 桌面作業系統從一個工作階段的下一步。 向使用者這個桌面會稍有不同於其他虛擬或實體裝置，除了它透過網路存取。

最佳化設定會參考裝置上進行。 VM 是一個理想位置來建置映像，因為您可以儲存的狀態，請檢查點，而且可以進行備份，以及其他有用的工作。 啟動基底 VM 上安裝預設的作業系統，並移除不必要的應用程式、 安裝 Windows 更新、 安裝其他更新、 刪除暫存檔案、 套用等設定，然後最佳化 VDI 用於基底 VM。

有其他類型的持續性和遠端桌面服務 (RDS) 等的 VDI。 有關這些技術的深入討論已超出本主題的重點放在 Windows 的基底參考主應用程式最佳化 」 等環境中的其他因素的映像設定的範圍。

### <a name="persistent-vdi"></a>持續 VDI

持續 VDI 是，在基本層級中，儲存之間重新啟動的作業系統狀態的 VM。 VDI 解決方案的其他軟體層會提供使用者容易且流暢地存取其指派的 vm，通常會使用單一登入解決方案。

有數個不同的實作，持續 VDI 的：

-   傳統的虛擬機器，其中 VM 有它自己的虛擬磁碟檔案，正常啟動、 將變更從一個工作階段儲存到下一步，和在本質上是只是一般的 VM。 差別在於使用者如何存取此 VM。 可能會有 web 入口網站使用者登入，會自動將使用者導向至其一個或多個指派 VDI Vm。

-   映像為基礎持續性虛擬機器，與個人虛擬磁碟。 在這種實作有一或多個主機伺服器上是基底/黃金映像。 建立 VM，然後建立並指派給此磁碟的永續性儲存體一或多個虛擬磁碟。

    -   啟動 VM 時，會將一份基底映像讀入記憶體的 VM。 在此同時，指派給該 VM 上，使用任何先前的作業系統變更的永續性虛擬磁碟合併完成複雜的程序。

    -   例如事件記錄檔寫入、 記錄檔寫入 」 等的變更會重新導向至 讀取/寫入虛擬磁碟，並指派到該 VM。

    -   在此情況下，作業系統和應用程式服務可能會正常運作，使用傳統服務的軟體，例如 Windows Server Update Services 或其他管理技術。

### <a name="non-persistent-vdi"></a>非持續 VDI

以基底或 「 黃金 」 的映像為基礎的非持續 VDI 實作，基底映像，然後再透過本機設定與本機原則，將大部分會執行最佳化。

映像為基礎的非持續 VDI 與基底映像會處於唯讀狀態。 非持續 VDI VM 啟動時，串流處理一份基底映像的 vm。 在啟動期間，直到下次重新開機之後，就會發生的活動會重新導向到暫存位置。 通常使用者提供網路位置來儲存其資料。 在某些情況下，使用者的設定檔會合併標準 VM，以提供使用者他們的設定。

其中一個重要層面以單一映像為基礎的非 persitent VDI 為提供服務。 將更新作業系統傳遞通常每月一次。
映像為基礎的 vdi，還有一組處理序執行，以取得映像的更新：

-   在指定的主機上，衍生自該主機上的所有 Vm 的基底映像必須關閉或已關閉。 這表示使用者會重新導向至其他 Vm。

-   然後開啟並啟動基底映像。 然後執行所有維護活動，例如作業系統更新、.NET 更新、 應用程式更新等。

-   要套用的任何新設定會套用在此階段中。

-   此時，會執行其他維護作業。

-   然後關閉基底映像。

-   基底映像是密封格式，且設定移至生產環境。

-   允許使用者重新登入。

> [!NOTE]  
> Windows 10 上定期執行，請以自動執行維護工作的一組。 沒有預設設定為在上午 3:00 當地時間每天執行排程的工作。 此排程的工作執行工作，包括 Windows Update 清除的清單。 您可以檢視所有維護工作會自動使用此 PowerShell 命令進行類別目錄：

`Get-ScheduledTask | ? {$_.Settings.MaintenanceSettings}`



使用非持續 VDI 的挑戰之一是當使用者登出時，幾乎所有的作業系統活動捨棄。 可能會儲存使用者的設定檔和/或狀態，但虛擬機器本身會捨棄上次開機之後所做的幾乎所有變更。 因此，不適用最佳化適用於 Windows 的電腦，將一個工作階段狀態儲存至下一步。

大小取決於 VM VDI 架構，like 預先擷取和 SuperFetch 不將有助於從一個工作階段的下一步，因為 VM 上的所有最佳化都會被都捨棄的項目重新啟動。 編製索引可能會部分浪費資源，例如傳統的磁碟重組的任何磁碟最佳化也是一樣。

### <a name="to-sysprep-or-not-sysprep"></a>Sysprep 或不 Sysprep

Windows 10 有內建功能，稱為[系統準備工具](https://docs.microsoft.com/windows-hardware/manufacture/desktop/sysprep--system-preparation--overview)，(通常 abbreviiated"Sysprep")。 Sysprep 工具來準備以重複使用自訂的 Windows 10 映像。 Sysprep 程序可確保產生的作業系統是正確唯一在生產環境中執行。

有與其執行 Sysprep 的原因。 在 VDI 的情況下，您可以自訂會做為登入使用此映像的後續使用者設定檔範本的預設使用者設定檔的能力。 您可能會有您要安裝，但也能夠控制每個應用程式設定的應用程式。

替代方法是使用標準。ISO 安裝，可能使用自動的安裝回應檔案，以及工作順序以安裝應用程式，或移除應用程式。 您也可以使用 工作順序，可能使用在圖中，設定本機原則設定[本機群組原則物件公用程式 (LGPO)](https://blogs.technet.microsoft.com/secguide/2016/01/21/lgpo-exe-local-group-policy-object-utility-v1-0/)工具。

### <a name="universal-windows-platform-app-cleanup"></a>通用 Windows 平台應用程式清除

其中一個 VDI 映像的目標是要越小越好。 移除在環境中的 UWP 應用程式，將不會使用為單向，以減少映像的大小。 使用 UWP 應用程式，有主要的應用程式檔案，也就是裝載。 沒有少量的資料儲存在應用程式特定設定的每個使用者的設定檔。 另外還有少量的所有使用者設定檔中的資料。

連線時間等所有項目談到 UWP 應用程式清除。 如果您將基底映像部署到其中一個裝置沒有網路連線時，Windows 10 無法連接到 Microsoft Store 並下載應用程式和安裝它們，而您嘗試解除安裝後再試。

如果您修改您的基底。WIM 您用來安裝 Windows 10，並移除不必要的 UWP 應用程式。之前您安裝、 應用程式不會開始安裝，而且您的設定檔的建立時間應該短的 WIM。 稍後在本節中，您將找到有關如何從您的安裝移除 UWP 應用程式。WIM 檔案。

VDI 是不錯的策略是佈建您想要在基底映像，然後限制或封鎖 Microsoft Store 的存取之後的應用程式。 市集應用程式會定期更新，在一般的電腦上的背景。 UWP 應用程式可以更新維護期間，套用其他更新的時間。 

#### <a name="delete-the-payload-of-uwp-apps"></a>刪除 UWP 應用程式的裝載

UWP 應用程式不需要是仍在檔案系統中耗用少量的磁碟空間。 針對永遠不會將所需的應用程式，可以使用 PowerShell 命令的基底映像移除不必要的 UWP 應用程式的承載。

事實上，如果您移除安裝。使用連結的 WIM 檔案稍後將提供在本節中，您應該能夠從頭開始非常精簡的 UWP 應用程式清單。


### <a name="tuning-windows-10-network-performance-by-using-registry-settings"></a>微調的 Windows 10 的網路效能，使用登錄設定

這是在 VDI 或實體電腦已經是主要是基礎的網路工作負載的環境中特別重要。 此區段的偏差效能設定，到偏好網路，設定額外的緩衝處理和快取的項目，依此類推喜歡目錄項目。

請注意，某些設定，這一節*登錄為基礎只*和應將其納入基底映像之前用於生產環境部署映像。

下列設定值記載於[Windows Server 2016 效能微調指導方針](https://docs.microsoft.com/windows-server/administration/performance-tuning/)Windows 產品小組所發佈的 Microsoft.com 上的資訊。

#### <a name="disablebandwidththrottling"></a>DisableBandwidthThrottling

HKLM\\System\\CurrentControlSet\\Services\\LanmanWorkstation\\Parameters\\DisableBandwidthThrottling

適用於 Windows 10。 預設值是**0**。 根據預設，在某些情況下，SMB 重新導向器會對高延遲網路連線的輸送量進行節流，以避免網路相關逾時。 此登錄值設定為**1**停用此節流，在高延遲網路連線、 啟用更高的檔案傳輸輸送量，因此您應該考慮這項設定。

#### <a name="fileinfocacheentriesmax"></a>FileInfoCacheEntriesMax

HKLM\\System\\CurrentControlSet\\Services\\LanmanWorkstation\\Parameters\\FileInfoCacheEntriesMax

適用於 Windows 10。 預設值是**64**，有效的範圍為 1 到 65536。 此值用來決定用戶端可以快取的檔案中繼資料數量。 增加此值，可以減少網路流量，並提升效能，存取許多檔案時。 請嘗試增加此值可**1024年**。

#### <a name="directorycacheentriesmax"></a>DirectoryCacheEntriesMax

HKLM\\System\\CurrentControlSet\\Services\\LanmanWorkstation\\Parameters\\DirectoryCacheEntriesMax

適用於 Windows 10。 預設值是**16**，有效的範圍為 1 到 4096。 此值用來決定用戶端可以快取的目錄資訊數量。 提高此值可降低網路流量，並且在存取大量目錄時提升效能。 請考慮增加此值可**1024年**。

#### <a name="filenotfoundcacheentriesmax"></a>FileNotFoundCacheEntriesMax

HKLM\\System\\CurrentControlSet\\Services\\LanmanWorkstation\\Parameters\\FileNotFoundCacheEntriesMax

適用於 Windows 10。 預設值是**128**，有效的範圍為 1 到 65536。 此值用來決定用戶端可以快取的檔案名稱資訊數量。 增加此值，可以減少網路流量，並提升效能，存取許多的檔案名稱時。 請考慮增加此值可**2048年**。

#### <a name="dormantfilelimit"></a>DormantFileLimit

HKLM\\System\\CurrentControlSet\\Services\\LanmanWorkstation\\Parameters\\DormantFileLimit

適用於 Windows 10。 預設值是**1023年**。 此參數可指定在應用程式關閉檔案之後，共用資源上應該保持開啟的檔案數目上限。 在數千個用戶端要連接到 SMB 伺服器，請考慮減少此值可**256**。

您可以使用來設定這些 SMB 設定的許多[組 SmbClientConfiguration](https://docs.microsoft.com/powershell/module/smbshare/set-smbclientconfiguration)並[組 SmbServerConfiguration](https://docs.microsoft.com/powershell/module/smbshare/set-smbserverconfiguration) Windows PowerShell cmdlet。 您可以設定登錄專用的設定，使用 Windows PowerShell，如下列範例所示：

`Set-ItemProperty -Path "HKLM:\\SYSTEM\\CurrentControlSet\\Services\\LanmanWorkstation\\Parameters" RequireSecuritySignature -Value 0 -Force`

> [!NOTE]  
> [磁碟清理精靈] 不再正在開發中。 Windows 會使用其他方法，來提供磁碟清除函式。



以下是建議的各種磁碟清理工作。 您應該測試前先實作其中任何一項：

1.  在套用所有更新之後執行磁碟清理精靈 （提高權限）。 包含類別**傳遞最佳化**並**Windows Update 清除**。 您可以自動化此程序**Cleanmgr.exe**具有 **/SAGESET:11**選項。 這個選項會設定登錄值，可以稍後可用於自動執行磁碟清理工作，使用磁碟清理 精靈中的每個可用的選項。

    1.  測試 VM 上，從全新安裝，請執行**Cleanmgr.exe /SAGESET:11**會顯示有預設啟用的只有兩個自動磁碟清理選項：

    - 下載的程式檔案

    - Temporary Internet Files

    2.  如果您設定更多選項或所有選項，這些選項會在登錄中，記錄根據前一個命令中提供的索引值 (**Cleanmgr.exe /SAGESET:11**)。 在此範例中，我們使用值*11*為我們的索引，如需後續自動執行磁碟清理程序。

    3.  執行後**Cleanmgr.exe /SAGESET:11**您會看到一些的磁碟清理選項的分類。 您可以選取每個選項，然後按**確定**。 您會發現 [磁碟清理精靈]，就會消失。 不過，您選取的設定會儲存在登錄中，而且可以執行叫用**Cleanmgr.exe /SAGERUN:11**。

2.  如果任何使用中，請清除磁碟區陰影複製的儲存體。 若要這樣做，請在提升權限的提示字元中執行下列命令：

    - **vssadmin list shadows**

    - **Vssadmin 清單 shadowstorage**

        這些命令的輸出是否*找不到符合查詢。* ，則使用中沒有任何 VSS 儲存體。

3.  清除暫存檔案和記錄檔。 從提升權限的命令提示字元中，執行下列命令：

    - **Del c:\\\*.tmp /s**

    - **Del c:\\Windows\\Temp\\。**

    - **Del %temp%\\。**

4.  刪除任何未使用的設定檔，使用此命令在系統上：

    **wmic 路徑 win32_UserProfile 其中 LocalPath ="c:\\\\使用者\\\\\<使用者\>"刪除**

### <a name="remove-onedrive"></a>移除 OneDrive

移除 OneDrive 牽涉到移除封裝，解除安裝和移除\*.lnk 檔案。 您可以使用下列 PowerShell 程式碼範例，以協助從映像移除 OneDrive:

```azurecli

Taskkill.exe /F /IM "OneDrive.exe"
Taskkill.exe /F /IM "Explorer.exe"` 
    if (Test-Path "C:\\Windows\\System32\\OneDriveSetup.exe")`
     { Start-Process "C:\\Windows\\System32\\OneDriveSetup.exe"`
         -ArgumentList "/uninstall"`
         -Wait }
    if (Test-Path "C:\\Windows\\SysWOW64\\OneDriveSetup.exe")`
     { Start-Process "C:\\Windows\\SysWOW64\\OneDriveSetup.exe"`
         -ArgumentList "/uninstall"`
         -Wait }
Remove-Item -Path
"C:\\Windows\\ServiceProfiles\\LocalService\\AppData\\Roaming\\Microsoft\\Windows\\Start Menu\\Programs\\OneDrive.lnk" -Force
Remove-Item -Path "C:\\Windows\\ServiceProfiles\\NetworkService\\AppData\\Roaming\\Microsoft\\Windows\\Start Menu\\Programs\\OneDrive.lnk" -Force \# Remove the automatic start item for OneDrive from the default user profile registry hive
Start-Process C:\\Windows\\System32\\Reg.exe -ArgumentList "Load HKLM\\Temp C:\\Users\\Default\\NTUSER.DAT" -Wait
Start-Process C:\\Windows\\System32\\Reg.exe -ArgumentList "Delete HKLM\\Temp\\SOFTWARE\\Microsoft\\Windows\\CurrentVersion\\Run /v OneDriveSetup /f" -Wait
Start-Process C:\\Windows\\System32\\Reg.exe -ArgumentList "Unload HKLM\\Temp" -Wait Start-Process -FilePath C:\\Windows\\Explorer.exe -Wait
```
