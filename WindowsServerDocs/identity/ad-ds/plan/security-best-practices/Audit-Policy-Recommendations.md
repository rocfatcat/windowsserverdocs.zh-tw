## <a name="events-to-monitor"></a>事件監視器  
產生安全性警告完美的事件編號應包含下列屬性：  

-   很可能該發生表示未經授權的活動  

-   較少的誤判  

-   發生應該會變為調查日 forensics 回應  

兩種類型的事件應該會監視和提醒：  

1.  在這一次表示未經授權的活動的事件  

2.  事件預期和接受基準上方的累積  

第一次約會的範例如下：  

如果登入電腦不為網域控制站禁止網域系統管理員 (DAs)，執行一次登入的使用者工作站 DA 成員應該產生警示和調查。 這種類型的警示很容易使用稽核特殊登入事件 4964 產生 （特殊群組已指派給新的登入）。 其他的單一警示的範例包括：  

-   如果伺服器 B，當連接到彼此的警示不應該連接伺服器 A。  

-   如果一般使用者 account 意外加入機密安全性群組，警示。  

-   如果原廠位置 A 員工一律不運作晚上，警示時的使用者登入午夜。  

-   如果已安裝未經授權的服務網域控制站，提醒。  

-   如果一般的使用者想要直接登入的他們使用的是不清除理由這樣 SQL Server，調查。  

-   如果您有任何成員 DA 群組中，且其他發生新增本身，請立即檢查。

登入失敗的 aberrant 數字可能猜測攻擊密碼。 為企業提供警示大量登入失敗，就必須先了解登入失敗之前惡意的安全性事件它們的環境中的一般層級。

## <a name="general-list-of-security-event-id-recommendation-criticalities"></a>一般的安全性事件 ID 建議 Criticalities 的清單  
所有 263 建議都附帶關鍵性評等，如下所示：  

**高：**事件 Id 高嚴重性評等與務必立即會提醒及調查。  

**媒體：**中嚴重性評等與 263 可能表示惡意的活動，但必須伴隨一些其他 abnormality （例如，特殊號碼特定時段、 未預期的項目或通常會未預期事件登入的電腦上的項目中發生。）。 媒體-關鍵性事件可能也會收集為計量並隨著時間相較於 r。  

**低：**低嚴重性事件的事件編號應該不會獲得注意或造成警示，除非中或高嚴重性活動的相互關聯。  

這些建議是基準指南提供系統管理員。 應該完全之前實作 production 環境中檢視所有的建議。  

請參考[到監視器附錄 l： 事件](../../../ad-ds/plan/Appendix-L--Events-to-Monitor.md)針對一份建議的事件監視器，其關鍵性評等，並事件郵件摘要。
