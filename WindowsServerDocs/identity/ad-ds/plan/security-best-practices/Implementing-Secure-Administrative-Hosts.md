# <a name="implementing-secure-administrative-hosts"></a>實作安全的管理主機

>適用於：Windows Server 2016、Windows Server 2012 R2、Windows Server 2012

安全的系統管理主機是已特別針對建立安全平臺所設定的工作站或伺服器，而特殊許可權帳戶可以在 Active Directory 或網域控制站上執行系統管理工作。已加入網域的系統，以及在已加入網域的系統上執行的應用程式。 在此情況下，「特殊許可權帳戶」不僅指的是 Active Directory 中最具特殊許可權群組成員的帳戶，還會參考任何已委派許可權的帳戶，以允許執行系統管理工作。  
  
這些帳戶可能是支援服務中心帳戶，其能夠為網域中的大部分使用者重設密碼、用來管理 DNS 記錄和區域的帳戶，或用於設定管理的帳戶。 安全的系統管理主機專門用來管理功能，而且不會執行像是電子郵件應用程式、網頁瀏覽器或生產力軟體（例如 Microsoft Office）的軟體。  
  
雖然「最具特殊許可權的」帳戶和群組應該會適當地受到而更加嚴格保護，但這並不會讓您不需要保護任何帳戶和群組，而是已授與標準使用者帳戶的許可權。  
  
安全的系統管理主機可以是專用的工作站，僅用於系統管理工作、執行遠端桌面閘道伺服器角色的成員伺服器，以及 IT 使用者連線來執行目的地主機管理的伺服器，或是執行Hyper-v 角色並為每個 IT 使用者提供唯一的虛擬機器，以用於其系統管理工作。 在許多環境中，可能會實作為這三種方法的組合。  
  
執行安全的系統管理主機需要規劃和設定，與貴組織的大小、管理實務、風險胃口和預算一致。 這裡提供用來執行安全管理主機的考慮和選項，讓您在開發適合您組織的系統管理策略時使用。  
  
## <a name="principles-for-creating-secure-administrative-hosts"></a>建立安全的系統管理主機的原則  
若要有效保護系統免于遭受攻擊的安全，請記住一些一般原則：  
  
1.  您絕對不應該從較不受信任的主機（也就是未受保護的工作站與管理的系統相同的程度）管理信任的系統（也就是安全伺服器，例如網域控制站）。  
  
2.  執行特殊許可權活動時，您不應該依賴單一驗證因素;也就是說，使用者名稱和密碼組合不應視為可接受的驗證，因為只會表示單一要素（您知道的東西）。 您應該考慮在系統管理案例中，認證的產生和快取或儲存位置。  
  
3.  雖然目前威脅環境中的大部分攻擊都會利用惡意程式碼和惡意攻擊，但是在設計和執行安全的系統管理主機時，並不會省略實體安全性。  
  
### <a name="account-configuration"></a>帳戶設定  
即使您的組織目前未使用智慧卡，您仍應考慮為特殊許可權帳戶和安全的系統管理主機來實行。 系統管理主機應設定為需要所有帳戶的智慧卡登入，方法是修改連結至包含管理主機之 Ou 的 GPO 中的下列設定：  
  
**電腦設定 \ \windows \ 本機原則 \ Options\Interactive 登入：需要智慧卡**  
  
這項設定會要求所有互動式登入都使用智慧卡，而不論 Active Directory 中個別帳戶的設定為何。  
  
您也應該將安全的系統管理主機設定為只允許授權帳戶的登入，這可以在中進行設定：  
  
**電腦設定 \ \windows \ 本機原則 \ 本機原則 \ 許可權指派**  
  
這只會授與互動式（且適當的遠端桌面服務）登入許可權給安全管理主機的授權使用者。  
  
### <a name="physical-security"></a>實體安全性  
若要讓系統管理主機被視為值得信任，必須設定和保護其與管理的系統相同的程度。 [保護網域控制站免于遭受攻擊](../../../ad-ds/plan/security-best-practices/Securing-Domain-Controllers-Against-Attack.md)所提供的大部分建議也適用于用來管理網域控制站和 AD DS 資料庫的主機。 在大部分環境中執行安全系統管理系統的其中一項挑戰是，實體安全性可能較難執行，因為這些電腦通常位於與資料中心內裝載的伺服器不安全的區域中，例如系統管理使用者的桌上型電腦。  
  
實體安全性包括控制系統管理主機的實體存取。 在小型組織中，這可能表示您維護專用的系統管理工作站，在辦公室或桌上不使用時保持鎖定。 或者，這可能表示當您需要執行 Active Directory 或網域控制站的管理時，您可以直接登入網域控制站。  
  
在中型組織中，您可以考慮執行位於辦公室安全位置的安全系統管理「跳躍伺服器」，並在需要管理 Active Directory 或網域控制站時使用。 您也可以在不使用的情況下，執行在安全位置鎖定的系統管理工作站，不論是否有跳躍伺服器。  
  
在大型組織中，您可以部署資料中心存放的跳躍伺服器，提供嚴格控制的 Active Directory 存取權;網域控制站;和檔案、列印或應用程式伺服器。 執行跳躍伺服器架構最有可能包含安全工作站和伺服器在大型環境中的組合。  
  
無論您的組織大小和系統管理主機的設計為何，您都應該保護實體電腦免于未經授權的存取或遭竊，並且應該使用 BitLocker 磁碟機加密來加密和保護系統管理主機上的磁片磁碟機. 藉由在系統管理主機上執行 BitLocker，即使主機遭竊或其磁片已被移除，您仍可以確保未經授權的使用者無法存取磁片磁碟機上的資料。  
  
### <a name="operating-system-versions-and-configuration"></a>作業系統版本和設定  
所有的系統管理主機（不論是伺服器或工作站），都應該根據本檔稍早所述的原因，執行貴組織使用的最新作業系統。 藉由執行目前的作業系統，您的系統管理人員將受益于新的安全性功能、完整的廠商支援，以及作業系統中引進的其他功能。 此外，當您在評估新的作業系統時，先將它部署到系統管理主機上，您必須熟悉它所提供的新功能、設定和管理機制，之後就可以利用它來規劃更廣泛的作業系統部署。 然後，貴組織中最複雜的使用者也會是熟悉新作業系統的使用者，而且最適合用來支援它。  
  
### <a name="microsoft-security-configuration-wizard"></a>Microsoft 安全性設定精靈  
如果您在系統管理主機策略中執行跳躍伺服器，就應該使用內建的安全性設定向導來設定服務、登錄、audit 和防火牆設定，以減少伺服器的受攻擊面。 當您收集並設定 [安全性設定] Wizard 設定時，可以將設定轉換成用來在所有跳躍伺服器上強制執行一致基準設定的 GPO。 您可以進一步編輯 GPO 來執行跳躍伺服器特定的安全性設定，並且可以結合所有設定與從 Microsoft 安全性合規性管理員解壓縮的其他基準設定。  
  
### <a name="microsoft-security-compliance-manager"></a>Microsoft Security Compliance Manager  
[Microsoft 安全性合規性管理員](https://technet.microsoft.com/library/cc677002.aspx)是一項免費的工具，可整合 Microsoft 建議的安全性設定（根據作業系統版本和角色設定），並以單一工具和 UI 收集，以用於建立和設定網域控制站的基準安全性設定。 Microsoft 安全性合規性管理員範本可以與 [安全性設定] [Wizard] 設定結合，針對部署在跳躍伺服器所在 Ou 的 Gpo 中部署和強制執行的「跳躍伺服器」，產生完整的設定基準。位於 Active Directory。  
  
> [!NOTE]  
> 在撰寫本文時，Microsoft 安全性合規性管理員不會包含跳躍伺服器或其他安全系統管理主機的特定設定，但仍可使用安全性合規性管理員（SCM）來建立系統管理的初始基準主機. 不過，若要適當地保護主機，您應該將額外的安全性設定套用到高度安全的工作站和伺服器。  
  
### <a name="applocker"></a>AppLocker  
系統管理主機和虛擬 machinesshould 透過 AppLocker 或協力廠商應用程式限制軟體，透過腳本、工具和應用程式白名單進行設定。 任何未遵守安全設定的系統管理應用程式或公用程式，都應該升級或取代為符合安全開發和管理實務的工具。 當系統管理主機需要新的或額外的工具時，應該徹底測試應用程式和公用程式，而且如果工具適合部署在系統管理主機上，則可以將它新增至系統的白名單中。  
  
### <a name="rdp-restrictions"></a>RDP 限制  
雖然特定設定會根據系統管理系統的架構而有所不同，但您應該包括哪些帳戶和電腦可以用來建立與受控系統遠端桌面通訊協定（RDP）連線的限制，例如，使用遠端桌面閘道（RD 閘道）跳躍伺服器，從授權的使用者和系統控制網域控制站和其他受管理系統的存取權。  
  
您應該允許經過授權的使用者進行互動式登入，而且應該移除或甚至封鎖伺服器存取不需要的其他登入類型。  
  
### <a name="patch-and-configuration-management"></a>修補和設定管理  
較小的組織可能會依賴 Windows Update 或[Windows Server Update Services](https://technet.microsoft.com/windowsserver/bb332157) （WSUS）之類的供應專案來管理 Windows 系統更新的部署，而較大型的組織可能會實行企業修補程式和設定管理軟體，例如 Microsoft 端點 Configuration Manager。 無論您用來將更新部署至一般伺服器和工作站擴展的機制為何，您都應該考慮針對高度安全的系統（例如網域控制站、憑證授權單位單位和管理主機）進行個別部署。 藉由將這些系統與一般管理基礎結構隔離，如果您的管理軟體或服務帳戶遭到入侵，就無法輕易地將危害延伸到基礎結構中最安全的系統。  
  
雖然您不應該為安全系統執行手動更新程式，但您應該設定不同的基礎結構來更新安全系統。 即使在非常大型的組織中，此基礎結構通常也可以透過專用的 WSUS 伺服器和 Gpo 來執行，以保護系統。  
  
### <a name="blocking-internet-access"></a>封鎖網際網路存取  
系統管理主機不應允許存取網際網路，也不應該能夠流覽組織的內部網路。 系統管理主機上不應允許 Web 瀏覽器和類似的應用程式。 您可以透過安全主機上的周邊防火牆設定、WFAS 設定和「黑洞」 proxy 設定的組合，封鎖安全主機的網際網路存取。 您也可以使用應用程式允許清單，防止在系統管理主機上使用網頁瀏覽器。  
  
### <a name="virtualization"></a>虛擬化  
可能的話，請考慮將虛擬機器做為系統管理主機來執行。 使用虛擬化，您可以建立集中儲存和管理的每個使用者系統管理系統，並在不使用時輕鬆地關閉，以確保系統管理系統上的認證不會保持作用中狀態。 您也可以要求在每次使用之後，將虛擬系統管理主機重設為初始快照集，以確保虛擬機器維持初始化。 下一節會提供管理主機虛擬化選項的詳細資訊。  
  
## <a name="sample-approaches-to-implementing-secure-administrative-hosts"></a>執行安全管理主機的範例方法  
無論您設計和部署系統管理主機基礎結構的方式為何，都應該記住本主題前面「建立安全管理主機的原則」中提供的指導方針。 此處所述的每一種方法，都能提供有關如何將 IT 人員所使用的「系統管理」和「生產力」系統分開的一般資訊。 生產力系統是 IT 系統管理員用來檢查電子郵件、流覽網際網路，以及使用一般生產力軟體（例如 Microsoft Office）的電腦。 系統管理系統是強化的電腦，專門用來進行 IT 環境的日常管理。  
  
執行安全的系統管理主機最簡單的方式，就是為您的 IT 人員提供安全的工作站，讓他們能夠執行系統管理工作。 在僅限工作站的執行中，每個系統管理工作站都用來啟動管理工具和 RDP 連線，以管理伺服器和其他基礎結構。 僅限工作站的執行可在較小的組織中生效，雖然較大型、更複雜的基礎結構可能受益于使用專用管理伺服器和工作站之系統管理主機的分散式設計，如下所示：本主題稍後的「執行安全的系統管理工作站和跳躍伺服器」中所述。  
  
### <a name="implementing-separate-physical-workstations"></a>執行個別的實體工作站  
您可以執行系統管理主機的其中一種方式，就是將每個 IT 使用者都發佈到兩部工作站。 其中一個工作站與「一般」使用者帳戶搭配使用，以執行像是檢查電子郵件和使用產能應用程式等活動，而第二個工作站則完全專用於系統管理功能。  
  
對於生產力工作站，IT 人員可以指定一般使用者帳戶，而不是使用特殊許可權帳戶來登入不安全的電腦。 系統管理工作站應該以而更加嚴格控制的設定進行設定，而 IT 人員應該使用不同的帳戶來登入系統管理工作站。  
  
如果您已執行智慧卡，應將系統管理工作站設定為需要智慧卡登入，而且應該為 IT 人員提供不同的帳戶以供管理使用，同時也設定為需要智慧卡進行互動式登入。 系統管理主機應依照先前的說明進行強化，而且只有指定的 IT 使用者可以在本機登入管理工作站。  
  
#### <a name="pros"></a>優點  
藉由執行個別的實體系統，您可以確保每一部電腦都已針對其角色適當設定，而且 IT 使用者不會不慎暴露系統管理系統的風險。  
  
#### <a name="cons"></a>缺點  
  
-   執行個別的實體電腦會增加硬體成本。  
  
-   使用用來管理遠端系統的認證登入實體電腦，會將認證快取到記憶體中。  
  
-   如果未安全地儲存系統管理工作站，它們可能會透過實體硬體按鍵記錄器或其他實體攻擊等機制受到危害。  
  
### <a name="implementing-a-secure-physical-workstation-with-a-virtualized-productivity-workstation"></a>使用虛擬化的生產力工作站來執行安全的實體工作站  
在這種方法中，IT 使用者會獲得一個安全的系統管理工作站，讓他們能夠執行日常管理功能，使用遠端伺服器管理工具（RSAT）或 RDP 連線來連接到其責任範圍內的伺服器。 當 IT 使用者需要執行生產力工作時，他們可以透過 RDP 連線到以虛擬機器的形式執行的遠端生產力工作站。 每個工作站都應該使用個別的認證，而且應該執行智慧卡之類的控制項。  
  
#### <a name="pros"></a>優點  
  
-   系統管理工作站和生產力工作站會分開。  
  
-   使用安全工作站連接到生產力工作站的 IT 人員可以使用不同的認證和智慧卡，而特殊許可權認證則不會存放在較不安全的電腦上。  
  
#### <a name="cons"></a>缺點  
  
-   執行解決方案需要設計和執行工作，以及健全的虛擬化選項。  
  
-   如果實體工作站未安全地儲存，可能會受到危害硬體或作業系統的實體攻擊，使其容易受到通訊攔截。  
  
### <a name="implementing-a-single-secure-workstation-with-connections-to-separate-productivity-and-administrative-virtual-machines"></a>透過連線來執行單一安全工作站，以個別的「生產力」和「系統管理」虛擬機器  
在此方法中，您可以向 IT 使用者發出先前所述的單一實體工作站，並在哪些使用者沒有特殊許可權存取的情況下進行鎖定。 您可以為裝載于專用伺服器上的虛擬機器提供遠端桌面服務連線，為 IT 人員提供一個執行電子郵件和其他生產力應用程式的虛擬機器，以及另一個設定為使用者的虛擬機器專用的系統管理主機。  
  
對於虛擬機器，您應該使用用來登入實體電腦的帳戶以外的不同帳戶，來要求智慧卡或其他多重要素登入。 當 IT 使用者登入實體電腦之後，他們就可以使用其生產力智慧卡連線到遠端生產力電腦，以及個別的帳戶和智慧卡來連線到其遠端系統管理電腦。  
  
#### <a name="pros"></a>優點  
  
-   IT 使用者可以使用單一實體工作站。  
  
-   藉由針對虛擬主機要求不同的帳戶，並使用虛擬機器的遠端桌面服務連線，就不會在本機電腦的記憶體中快取 IT 使用者的認證。  
  
-   實體主機的安全性與管理主機的程度相同，因而降低了本機電腦遭到入侵的可能性。  
  
-   在 IT 使用者的生產力虛擬機器或其系統管理虛擬機器可能遭到入侵的情況下，虛擬機器很容易就會重設為「已知良好」狀態。  
  
-   如果實體電腦遭到入侵，就不會在記憶體中快取任何特殊許可權認證，而且使用智慧卡可以防止擊鍵記錄器洩露認證。  
  
#### <a name="cons"></a>缺點  
  
-   執行解決方案需要設計和執行工作，以及健全的虛擬化選項。  
  
-   如果實體工作站未安全地儲存，可能會受到危害硬體或作業系統的實體攻擊，使其容易受到通訊攔截。  
  
### <a name="implementing-secure-administrative-workstations-and-jump-servers"></a>執行安全的系統管理工作站和跳躍伺服器  
除了保護系統管理工作站或結合它們之外，您還可以執行安全跳躍伺服器，而系統管理使用者可以使用 RDP 和智慧卡連線到跳躍伺服器，以執行系統管理工作。  
  
跳躍伺服器應該設定為執行遠端桌面閘道角色，讓您能夠對跳躍伺服器和將從它管理的目的地伺服器執行限制。 可能的話，您也應該安裝 Hyper-v 角色，並建立[個人虛擬桌面](https://technet.microsoft.com/library/dd759174.aspx)或其他每一使用者的虛擬機器，以供系統管理使用者用於跳躍伺服器上的工作。  
  
藉由在跳躍伺服器上提供系統管理使用者的每個使用者虛擬機器，您可以為系統管理工作站提供實體安全性，而且系統管理使用者可以在未使用時重設或關閉其虛擬機器。 如果您不想要在相同的系統管理主機上安裝 Hyper-v 角色和遠端桌面閘道角色，您可以將它們安裝在不同的電腦上。  
  
盡可能使用遠端系統管理工具來管理伺服器。 遠端伺服器管理工具（RSAT）功能應該安裝在使用者的虛擬機器上（如果您不是要執行系統管理的每一使用者虛擬機器，則是跳躍伺服器），而且系統管理人員應該透過 RDP 連線到其執行系統管理工作的虛擬機器。  
  
在系統管理使用者必須透過 RDP 連線到目的地伺服器以直接管理的情況下，RD 閘道應該設定為只有在使用適當的使用者和電腦來建立與目的地的連線時，才允許建立連接。伺服器. 在未指定管理系統的系統上（例如一般用途的工作站和非跳躍伺服器的成員伺服器），應該禁止執行 RSAT （或類似的）工具。  
  
#### <a name="pros"></a>優點  
  
-   建立跳躍伺服器可讓您將特定伺服器對應至您網路中的「區域」（具有類似設定、連線和安全性需求的系統集合），並要求系統管理人員可以達成每個區域的管理從安全的系統管理主機連接到指定的「區域」伺服器。  
  
-   藉由將跳躍伺服器對應至區域，您可以針對連線屬性和設定需求執行細微的控制，並可輕鬆地識別從未經授權的系統連線的嘗試。  
  
-   藉由在跳躍伺服器上執行每個系統管理員的虛擬機器，您可以在管理工作完成時，強制關閉虛擬機器並將其重設為已知的清除狀態。 當系統管理工作完成時，強制關閉（或重新開機）虛擬機器，虛擬機器就無法成為攻擊者的目標，也不會有認證竊取攻擊，因為記憶體快取的認證不會保存超過重新開機的範圍。  
  
#### <a name="cons"></a>缺點  
  
-   無論是實體或虛擬，跳躍伺服器都需要專用伺服器。  
  
-   若要執行指定的跳躍伺服器和系統管理工作站，則需要仔細規劃和設定，以對應至環境中設定的任何安全性區域。
