# <a name="identifying-the-deployment-project-participants"></a>檢測軍人部署專案參與者

在 Active Directory Domain 服務 (AD DS) 建立部署專案的第一個步驟是建立設計和部署專案小組負責管理設計階段和階段專案 Active Directory 部署的循環。 此外，您必須找出的個人和負責使用與維護 directory 部署完畢之後群組。

## <a name="BKMK_1"></a>定義特定專案角色  
找出的人都是以角色特定專案是重要步驟建立團隊的專案。 其中包括贊助、 專案設計師，以及專案經理。 執行 Active Directory 部署專案負責這些人。  
  
專案架構和專案經理，您將之後，這些人建立的公司的通訊通道、 組建專案排程，並找出的人都將會開始使用各種不同的擁有者專案團隊的成員。  
  
### <a name="executive-sponsor"></a>贊助  
部署的基礎結構，例如 AD DS 可以影響廣泛的組織。 基於這個原因，會有贊助了部署商務值，支援在執行層級，專案及可協助您組織衝突。  
  
### <a name="project-architect"></a>專案架構  
Active Directory 部署的每個專案需要專案設計師管理 Active Directory 設計和部署決策程序。 設計師提供技術的專業，以協助您的設計和部署 AD DS 程序。  
  
> [!NOTE]  
> 如果您在組織中的不現有人員 directory 設計體驗，您可能想要雇用外顧問是一位專家 Active Directory 設計和部署。  
  
Active Directory 專案設計師責任包含下列類型：  
  
-   擁有 Active Directory 設計  
  
-   了解及錄製按鍵設計決策理由  
  
-   確認設計符合企業需求的組織  
  
-   建立共識設計、 部署、 之間作業小組  
  
-   了解 AD DS 整合應用程式的需求  
  
Active Directory 的最終設計，必須反映業務務目標和技術決策的組合。 因此，專案設計師必須檢視設計決策確保它們配合的業務務目標。  
  
### <a name="project-manager"></a>專案經理  
專案經理幫助您透過商務用單位和技術管理群組之間的合作。 理想、 Active Directory 部署的專案經理是其他人熟悉的兩個操作原則，IT 群組和部署 AD DS 準備群組的設計需求在組織中的人。 專案經理開頭設計和實作，透過持續監督整個部署專案，並確保專案保持排程與預算。 專案經理責任包含下列類型：  
  
-   提供基本專案，例如 「 排程和預算計劃  
  
-   在 Active Directory 設計和部署專案駕駛進度  
  
-   確保的適當的每個人都參與每個部分的設計程序  
  
-   做為 Active Directory 部署專案連絡人單點  
  
-   建立設計、 部署及作業團隊間通訊  
  
-   建立及維護贊助整個部署專案與通訊  
  
## <a name="BKMK_2"></a>建立者和系統管理員  
在 Active Directory 部署專案中，擁有者的人保持負責管理以確定該部署工作完成和規格需求組織該 Active Directory 設計。 擁有者不一定進行存取，或直接操作 directory 基礎結構。 系統管理員是負責完成所需的部署工作人員。 系統管理員可以網路存取及管理 directory 和其基礎結構所需的權限。  
  
擁有者的角色是策略與管理。 擁有者負責通訊的系統管理員所需的例如建立新的網域控制站樹系的 Active Directory 設計實作的工作。 將設計實作設計規格網路上的系統管理員負責。  
  
大型的組織中其他人填滿擁有者和系統管理員角色。不過，某些組織小，相同個人可能會做為擁有者和系統管理員。  
  
### <a name="service-and-data-owners"></a>服務與資料擁有者  
管理日常 AD DS 包含兩種類型的擁有者：  
  
-   負責計劃與長期維護 Active Directory 基礎結構和確保 directory 會繼續運作，且會保留 [建立服務層級合約中的目標服務擁有者  
  
-   負責的資訊儲存在 directory 維護資料擁有者。 這包括使用者電腦 account 管理及管理本機資源，例如成員伺服器及工作站。  
  
請務必，讓他們可以參與度設計程序的早期辨識 Active Directory 服務，資料擁有者。 因為服務和資料擁有負責長期維護 directory 部署專案完成後，請務必提供有關組織需要輸入，以及熟悉方式與原因特定設計的決策這些人的。 服務擁有者包括樹系擁有者、 Active Directory Domain 命名系統 (DNS) 擁有者，以及拓撲擁有者的網站。 資料擁有者包含單位 （組織單位） 擁有者。  
  
### <a name="service-and-data-administrators"></a>服務及資料的系統管理員  
操作 AD ds 包含兩種類型的系統管理員： 管理員和管理員資料服務。 服務管理員實作原則的決策服務擁有者和處理維護 directory 服務和基礎結構相關聯的日常工作。 這包括管理裝載 directory 服務，管理其他網路服務，例如所需的 AD DS 控制樹系設定，並確保 directory 都可使用 DNS 網域控制站。  
  
服務系統管理員負責也完成初始 Windows Server 2008 Active Directory 部署程序完成之後所需執行 Active Directory 部署工作。 例如 directory 增加需求，為中服務系統管理員建立其他網域控制站和建立或視移除之間網域信任。 基於這個原因，Active Directory 部署小組需要包括服務系統管理員。  
  
您必須小心指派給受信任的人員在組織中的服務的系統管理員角色。 這些人可以修改網域控制站系統檔案，因為它們可以變更 AD DS 的行為。 您必須先確定您在組織中的服務系統管理員，是熟悉操作的人而安全性原則，會在您的網路上的位置，並人員了解需要執行的原則。  
  
資料的系統管理員的使用者網域中有義務維護資料儲存在 AD DS，例如帳號使用者和群組和維護成員，他們網域的電腦都。 資料管理員控制 directory 物件的子集並無法安裝或 directory 服務設定所控制。  
  
預設不提供資料的系統管理員帳號。 設計團隊判斷要組織管理的資源將會如何之後，網域擁有必須建立資料系統管理員帳號，並委派它們根據設定的系統管理員的責任物件的適當權限。  
  
最好數量服務確保基礎結構繼續運作所需的最低號碼組織中的系統管理員。 大部分的系統管理工作資料系統管理員可以完成。 服務管理員需要設定，因為有義務維護 directory 和支援的基礎結構多少寬技術。 資料的系統管理員只需要管理其部分 directory 所需的技能。 將這種方式工作指派導致節省成本組織，因為只有少數系統管理員必須訓練運作及維護整個 directory 和其基礎結構。  
  
例如，服務系統管理員必須了解如何新增至樹系的網域。 這包括安裝的軟體来轉換的網域控制站伺服器以及管理 DNS 環境，使網域控制站可以自動接手合併 Active Directory 環境中。 了解如何管理他們所負責新帳號建立新的員工在他們部門例如的特定資料只需要資料系統管理員。  
  
部署 AD DS 需要協調與許多不同群組的網路基礎結構參與間通訊。 這些群組應該將服務，資料負責表示不同群組的設計和部署程序期間擁有者。  
  
部署專案完成之後，這些服務和資料擁有繼續負責的部分受其群組的基礎結構。 在 Active Directory 環境中，這些擁有者是樹系擁有者、 AD DS 擁有者 DNS、 網站拓撲擁有者和組織單位擁有者。 這些服務和資料擁有的角色所述的下列各節。  
  
#### <a name="forest-owner"></a>樹系擁有者  
樹系擁有者通常是誰負責 Active Directory 部署程序，而且人員最終負責部署完成之後維持樹系的服務傳遞組織中的資深資訊技術 (IT) 經理。 樹系擁有者指派個人填入的擁有權角色檢測軍人負責的人員在組織中可以提供所需資訊網路基礎結構和管理的需求。 樹系擁有者負責下列動作：  
  
-   建立樹森林根網域的部署  
  
-   每個來建立所需的樹系的網域網域中的第一個網域控制站的部署  
  
-   服務之子-森林所有網域中的系統管理員群組成員資格  
  
-   建立組織單位結構每個網域森林中的設計的  
  
-   委派給組織單位擁有者系統管理員權限  
  
-   變更架構  
  
-   樹系設定的變更  
  
-   實作的特定群組原則原則設定，包括網域使用者 account 原則，例如精細的密碼，以及 account 鎖定原則  
  
-   商務用原則設定可套用至網域控制站  
  
-   任何其他網域層級會套用群組原則設定  
  
樹系的擁有者授權單位上整個樹系。 負責森林擁有者的群組原則和商務原則設定，並選擇服務的系統管理員的人。 樹系擁有者為服務擁有者。  
  
#### <a name="dns-for-ad-ds-owner"></a>DNS AD DS 擁有者  
AD DS 擁有者 DNS 是深入瞭解現有的 DNS 基礎結構和組織的現有命名空間已經的人。  
  
AD DS 擁有者 DNS 負責下列動作：  
  
-   做為生意設計團隊之間目前擁有 DNS 基礎結構 IT 群組  
  
-   提供協助您建立新的 Active Directory 命名空間以組織相關現有的 DNS 名稱區資訊  
  
-   使用 deployment 團隊確認設計團隊的規格部署新的基礎結構 DNS 時，正常運作  
  
-   管理適用於 AD DS 基礎結構，包括的 DNS 伺服器服務和 DNS 資料 DNS  
  
AD DS 擁有者 DNS 是服務擁有者。  
  
#### <a name="site-topology-owner"></a>網站拓撲擁有者  
網站拓撲擁有者為熟悉實際句字的結構組織網路，包括個人子網路、 路由器和連接透過保守型連結網路區域的對應。 擁有者的網站拓撲負責下列動作：  
  
-   了解拓撲實體網路，以及其如何影響 AD DS  
  
-   了解如何在 Active Directory 部署會影響網路  
  
-   判斷要建立的 Active Directory 邏輯網站  
  
-   子網路新增時，更新網站物件的網域控制站修改，或移除  
  
-   建立網站連結、 網站連結橋接器，並手動連接物件  
  
網站拓撲擁有者為服務擁有者。  
  
#### <a name="ou-owner"></a>組織單位擁有者  
組織單位擁有者負責管理 directory 中儲存資料。 此個人必須熟悉操作，並在網路上的安全性原則。 組織單位擁有者只能執行這些工作，有服務系統管理員，委派給他們，他們只能執行這些工作 Ou 指派給上。 可能會指派給該組織單位擁有者的工作如下：  
  
-   執行他們指派組織單位所有 account 管理工作  
  
-   工作站和他們指派組織單位成員成員伺服器管理  
  
-   委派他們指派組織單位在本機系統管理員權限  
  
組織單位擁有者為資料擁有者。  
  
## <a name="BKMK_3"></a>建置專案小組  
Active Directory 專案團隊會暫時負責完成 Active Directory 設計和部署工作群組。 Active Directory 部署專案完成時，擁有者假設負責 directory，並可以遣散團隊的專案。  
  
專案團隊的大小，而有所不同組織的大小。 小型組織，可以責任團隊的專案的實體鍵盤保護蓋並參與部署的多個階段一個人。 大型的組織可能需要較大的小組，以不同的個人或甚至不同的小組涵蓋的責任的不同領域工作。 只要指派所有區域的責任，且符合組織的設計目的不是重要團隊的大小。  
  
### <a name="identifying-potential-forest-owners"></a>找出潛在的樹系擁有者  
找出您組織中的群組擁有及網路上的使用者提供 directory 服務所需的資源的控制項。 這些群組視為潛在的樹系擁有者。  
  
分開的服務，資料管理 AD DS 可以讓 IT 群組 （或群組） 的組織管理 directory 服務時，每個群組中的本機系統管理員管理自己的群組所屬的資料的基礎結構。 潛在的樹系擁有擁有所需的授權單位上部署及支援 AD DS 網路基礎結構。  
  
為組織打造基礎結構 IT 群組一個已 IT 群組則通常樹系擁有者，因此，任何未來部署潛在的樹系擁有者。 包含許多組織獨立基礎結構 IT 群組有許多潛在的樹系擁有者。 如果您的組織已經有 Active Directory 基礎結構就地，任何目前的樹系擁有者也有新部署的潛在的樹系擁有者。  
  
選取其中一個做為您考慮部署的每個樹系的樹系擁有者潛在的樹系擁有者。 這些潛在的樹系擁有負責處理設計團隊，以判斷他們的樹系將確實部署，或如果替代動作 （例如加入另一個現有的樹系） 是一種更佳的使用可用資源的並仍然符合他們的需求。 您在組織中的樹系擁有者 （或擁有者） 的 Active Directory 設計小組的成員。  
  
### <a name="establishing-a-design-team"></a>建立設計團隊  
Active Directory 設計團隊負責為收集有關 Active Directory 邏輯結構設計做出所需的所有資訊。  
  
設計團隊責任包含下列類型：  
  
-   判斷多少森林與網域和關聯的樹系之間的網域  
  
-   使用資料擁有者以確認設計符合他們安全性與管理的需求  
  
-   使用目前的網路系統管理員，以確保目前網路基礎結構支援設計和的設計將不會影響現有部署在網路上的應用程式  
  
-   若要確認設計符合建立的安全性原則組織的安全性群組的人員使用  
  
-   設計組織單位結構允許適當的保護層級和授權的適當委派給資料擁有者  
  
-   使用 deployment 測試設計團隊實驗室環境，確定它是做為計劃和修改設計為所需發生的任何問題的地址  
  
-   建立一個網站拓撲設計時防止使用頻寬載符合樹系的複寫需求。 設計拓撲網站的相關詳細資訊，請查看[設計網站拓撲適用於 Windows Server 2008 AD DS](https://technet.microsoft.com/library/cc772013.aspx)。  
  
-   使用 deployment 團隊確認設計實作正確  
  
設計團隊包含下列成員：  
  
-   潛在的樹系擁有者  
  
-   專案架構  
  
-   專案經理  
  
-   建立及維護網路上的安全性原則負責的人  
  
邏輯結構設計程序期間設計團隊辨識其他擁有者。 必須開始這些人，為他們都會參與設計程序。 部署小組交部署專案之後，設計團隊負責作業確認設計實作正確的部署程序。 設計團隊也會根據意見反應的測試設計變更。  
  
### <a name="establishing-a-deployment-team"></a>建立部署團隊  
Active Directory 部署的小組負責測試及實作 Active Directory 邏輯結構設計。 其中包括下列工作：  
  
-   建立這些可模擬 production 環境測試環境  
  
-   利用建議的樹系和網域結構來確認它符合每個角色擁有者的目標實驗室測試設計  
  
-   開發無障礙和測試提出的設計，測試環境中的任何移轉案例  
  
-   確認您的每個擁有者登出測試程序以確保測試正確的設計的功能  
  
-   測試試用的環境中的 deployment 作業  
  
設計與測試任務完成後，部署小組執行下列工作：  
  
-   建立的樹系和 Active Directory 邏輯結構設計根據網域  
  
-   網站做為所需的網站連結物件根據上建立的網站拓撲設計  
  
-   確保 DNS 基礎結構已支援 AD DS，而且的任何新命名空間已經整合組織的現有命名空間  
  
Active Directory 部署小組包含下列成員：  
  
-   樹系擁有者  
  
-   DNS AD DS 擁有者  
  
-   網站拓撲擁有者  
  
-   組織單位擁有者  
  
部署小組確保作業小組的成員全新的設計熟悉部署階段的運作方式的服務，資料系統管理員使用。 這有助於確保轉換的擁有權，在完成部署操作。 以完成部署程序，負責維護新的 Active Directory 環境會傳遞給作業小組。  
  
### <a name="documenting-the-design-and-deployment-teams"></a>設計和部署團隊文件  
文件名稱，並將參與部署 AD DS 與設計的人的聯絡資訊。 找出人員將會每個角色設計和部署的團隊負責。 一開始，此清單會包含潛在的樹系擁有、 專案經理，以及專案設計師。 當您判斷，將部署樹系的數目時，您可能需要額外的樹系建立新設計。 請注意，您將需要小組成員資格變更，以及當您在各種不同的 Active Directory 擁有者辨識設計程序期間，請更新您的文件。 協助您在擬設計和部署團隊每個樹系試算表，下載 Job_Aids_Designing_and_Deploying_Directory_and_Security_Services.zip 從工作協助工具的 Windows Server 2003 部署套件 ([https://go.microsoft.com/fwlink/?LinkID=102558](https://go.microsoft.com/fwlink/?LinkID=102558)) 以及開放「設計和部署小組資訊」(DSSLOGI_1.doc)。
