# 設定和管理仲裁

>適用於： Windows Server 2019、 Windows Server 2016、 Windows Server 2012 R2、 Windows Server 2012

本主題提供背景和步驟來設定及管理 Windows Server 容錯移轉叢集中的仲裁。

## 了解仲裁

在叢集中的仲裁取決於投票必須是作用中的叢集會員資格可正常啟動或繼續執行該叢集的一部分的元素數目。 如需更詳細的說明，請參閱[了解叢集與集區仲裁文件](../storage/storage-spaces/understand-quorum.md)的。

### 見證設定

一般規則是當您設定仲裁，叢集中的投票元素應該奇特的數字。 因此，如果叢集包含偶數投票節點，您應該設定磁碟見證或檔案共用見證。 叢集無法承受向下一個額外的節點。 此外，新增見證投票啟用繼續執行，如果半數的叢集節點同時向下，或已中斷連接的叢集。

如果所有節點都可以都看到磁碟，通常會建議磁碟見證。 當您需要考慮多站台的災害復原已複寫存放裝置與建議檔案共用見證。 可能的存放裝置廠商支援來自所有網站的讀寫存取來複寫的儲存體時，才會使用已複寫存放裝置設定磁碟見證。 <strong>*磁碟見證不支援使用儲存空間直接存取*</strong>。

### 節點投票指派

為進階的仲裁設定選項，您可以選擇要指派或移除仲裁票數，每個節點為基礎。 根據預設，所有節點會都指派投票。 無論投票指派的所有節點會繼續叢集中函式中，接收叢集資料庫更新，並可以裝載應用程式。

您可能會想要移除票數，從特定的災害復原組態中的節點。 例如，在多站台叢集中，您無法移除票數備份網站中的節點，讓這些節點不會影響仲裁計算。 此設定僅適用於手動容錯移轉的跨站建議。 如需詳細資訊，請參閱本主題中稍後的[仲裁考量的災害復原設定](#quorum-considerations-for-disaster-recovery-configurations)。

設定的節點的投票經過查閱叢集節點的**NodeWeight**常見屬性使用[Get ClusterNode](http://technet.microsoft.com/library/hh847268.aspx)Windows PowerShell cmdlet。 值為 0 表示節點沒有設定仲裁投票。 值為 1 表示節點的仲裁投票指派給，，它由叢集。 如需管理的節點投票的詳細資訊，請參閱本主題中稍後的[動態仲裁管理](#dynamic-quorum-management)。

使用**驗證叢集仲裁**驗證測試可驗證所有叢集節點的投票指派。

#### 節點投票指派的其他考量事項

  - 不建議節點投票指派強制執行的投票節點奇特的數目。 相反地，您應該設定磁碟見證或檔案共用見證。 如需詳細資訊，請參閱本主題中稍後的[見證設定](#witness-configuration)。
  - 如果已啟用動態仲裁管理，設定為已指派給節點票數節點可以有其票數指派或移除動態。 如需詳細資訊，請參閱本主題中稍後的[動態仲裁管理](#dynamic-quorum-management)。

### 動態仲裁管理

在 Windows Server 2012 中為進階的仲裁設定選項，您可以選擇啟用動態仲裁管理叢集。 如需如何動態仲裁運作方式的詳細資訊，請參閱[此說明](../storage/storage-spaces/understand-quorum.md#dynamic-quorum-behavior)。

使用動態仲裁管理也很可能會在最後一個留存叢集節點上執行的叢集。 以動態調整仲裁大部分需求，叢集可承受循序節點關機至單一節點。

使用[Get ClusterNode](https://docs.microsoft.com/powershell/module/failoverclusters/get-clusternode?view=win10-ps) Windows PowerShell cmdlet 的節點，叢集指派的動態投票可以驗證與叢集節點的**DynamicWeight**常見屬性。 值為 0 表示節點沒有仲裁投票。 值為 1 表示節點具有仲裁投票。

使用**驗證叢集仲裁**驗證測試可驗證所有叢集節點的投票指派。

#### 動態仲裁管理的其他考量事項

- 動態仲裁管理不允許承受投票成員的大部分同時發生失敗的叢集。 若要繼續執行，叢集永遠必須仲裁大部分在節點關機或失敗的階段。

- 如果您有明確移除節點的投票，叢集無法動態新增或移除該投票。
- 儲存空間直接存取啟用時，叢集只能支援兩個節點失敗。 這是[集區仲裁一節](../storage/storage-spaces/understand-quorum.md#poolQuorum)中深入說明

## 針對仲裁設定的一般建議

叢集軟體會自動設定為新的叢集，根據設定的節點數目和共用存放裝置的可用性仲裁。 這通常是最適合仲裁設定適用於該叢集。 不過，它會是不錯的想法，來建立叢集時，再放到生產環境的叢集之後，請檢閱仲裁設定。 若要檢視詳細的叢集仲裁設定，您可以使用 [驗證設定精靈] 或[測試叢集](https://docs.microsoft.com/powershell/module/failoverclusters/test-cluster?view=win10-ps)Windows PowerShell cmdlett，來執行**驗證仲裁設定**測試。 在容錯移轉叢集管理員中，基本仲裁設定會顯示在所選的叢集的摘要資訊，或您可以檢閱傳回當您執行 Windows PowerShell 的[Get ClusterQuorum](https://docs.microsoft.com/powershell/module/failoverclusters/get-clusterquorum?view=win10-ps)仲裁資源資訊cmdlet。

在任何時候，您可以執行**驗證仲裁設定**測試來驗證仲裁設定是最佳選擇您的叢集。 測試輸出會指出建議仲裁設定的變更，並設定，是最佳化。 如果建議變更，則您可以使用 [設定叢集仲裁精靈來套用建議的設定。

在生產環境中的叢集之後，請勿變更仲裁設定，除非您決定變更是適用於您的叢集。 您可能會想要考慮變更仲裁設定在下列情況：

- 新增或收回節點
- 新增或移除存放裝置
- 長期節點或見證失敗
- 復原多站台發生嚴重損壞修復案例中的叢集

如需有關如何驗證容錯移轉叢集的詳細資訊，請參閱[驗證容錯移轉叢集的硬體](<https://docs.microsoft.com/previous-versions/windows/it-pro/windows-server-2012-r2-and-2012/jj134244(v%3dws.11)>)。

## 設定叢集仲裁

您可以藉由使用容錯移轉叢集管理員或容錯移轉叢集的 Windows PowerShell cmdlet 設定叢集仲裁設定。

>[!IMPORTANT]
>它通常最好是使用仲裁設定的建議設定叢集仲裁精靈。 我們建議您決定變更是適用於您的叢集時，才自訂仲裁設定。 如需詳細資訊，請參閱本主題中的[仲裁設定的一般建議](#general-recommendations-for-quorum-configuration)。

### 設定叢集仲裁設定

在每個叢集的伺服器或對等項目上的本機**Administrators**群組成員資格是完成此程序所需的最低權限。 此外，您使用的帳戶必須是網域使用者帳戶。

>[!NOTE]
>您可以變更叢集仲裁設定，而不需停止的叢集或佔用叢集資源離線。

### 使用容錯移轉叢集管理員來變更容錯移轉叢集中的仲裁設定

1. 在容錯移轉叢集管理員中，選取，或指定您想要變更的叢集。
2. 選取，**動作**，在叢集選取**其他動作**，然後再選取**設定叢集仲裁設定**。 設定叢集仲裁精靈會出現。 選取 **\[下一步\]**。
3. 在**選取仲裁設定選項**頁面上，選取其中一個三個組態選項並完成該選項的步驟。 設定仲裁設定之前，您可以檢閱您的選擇。 如需有關選項的詳細資訊，請參閱本主題稍早的[容錯移轉叢集中的仲裁的概觀](#overview-of-the-quorum-in-a-failover-cluster)。

    - 若要允許自動重設對您目前的叢集設定是最佳的仲裁設定的叢集，選取 [**使用一般的設定**，然後完成精靈。
    - 若要新增或變更仲裁見證，選取 [**新增或變更仲裁見證**，然後完成下列步驟。 資訊及設定仲裁見證相關的考量，請參閱本主題稍早的[見證設定](#witness-configuration)。

      1. 在**選取仲裁見證**頁面上，選取一個選項來設定磁碟見證或檔案共用見證。 精靈會指出建議針對您的叢集見證選取項目選項。

          >[!NOTE]
          >您可以也選取 [**未設定仲裁見證**，然後完成精靈。 如果您有偶數投票您叢集中的節點，這可能會不是建議的設定。

      2. 如果您選取的選項，以設定磁碟見證，在**設定存放裝置見證**頁面上，選取您要指派成磁碟見證，存放磁碟區，然後完成精靈。
      3. 如果您選取選項來設定檔案共用見證，**設定檔案共用見證**頁面上，輸入或瀏覽至檔案共用作為見證資源，且然後完成精靈。

    - 若要設定仲裁管理設定和要新增或變更仲裁見證，選取**進階的仲裁設定和見證選取項目**，然後完成下列步驟。 資訊及關於進階的仲裁組態設定的考量，請參閱稍早在本主題中的[節點投票指派](#node-vote-assignment)和[動態仲裁管理](#dynamic-quorum-management)。

      1. 在**選取投票設定**頁面上，選取一個選項來指派給節點的投票。 根據預設，所有節點會都指派投票。 不過，針對特定案例中，您只能指派投票給節點的子集。

          >[!NOTE]
          >您也可以選取 [**否] 節點**。 這是通常不建議這樣做，因為它不允許節點可以參與仲裁投票，而它需要設定磁碟見證。 這個磁碟見證會變成單一失敗點的叢集。

      2. 在**設定仲裁管理**頁面上，您可以啟用或停用**來動態管理的節點指派允許叢集投票**選項。 選取這個選項通常會增加叢集的可用性。 預設會啟用該選項，並強烈建議不要停用此選項。 此選項可讓您繼續執行，此選項停用時，是不可能的失敗案例中的叢集。
      3. 在**選取仲裁見證**頁面上，選取一個選項來設定磁碟見證或檔案共用見證。 精靈會指出建議針對您的叢集見證選取項目選項。

          >[!NOTE]
          >您可以也選取 [**未設定仲裁見證**，然後完成精靈。 如果您有偶數投票您叢集中的節點，這可能會不是建議的設定。

      4. 如果您選取的選項，以設定磁碟見證，在**設定存放裝置見證**頁面上，選取您要指派成磁碟見證，存放磁碟區，然後完成精靈。
      5. 如果您選取選項來設定檔案共用見證，**設定檔案共用見證**頁面上，輸入或瀏覽至檔案共用作為見證資源，且然後完成精靈。

4. 選取 **\[下一步\]**。 確認您的選項會出現，確認頁面上，然後選取 [**下一步**。

精靈會執行，並**摘要**頁面隨即顯示，如果您想要檢視報告精靈執行的工作之後，選取 [**檢視報告**。 最新的報告將處於*systemroot * * * \\Cluster\\Reports** 名稱**QuorumConfiguration.mht**資料夾。

>[!NOTE]
>設定叢集仲裁之後，我們建議您執行**驗證仲裁設定**測試，確認已更新的仲裁設定。

### 對等的 Windows PowerShell 指令

下列範例顯示如何使用[Set-clusterquorum](https://docs.microsoft.com/powershell/module/failoverclusters/set-clusterquorum?view=win10-ps) cmdlet 與其他 Windows PowerShell cmdlet 設定叢集仲裁。

下列範例會變更仲裁設定叢集*CONTOSO FC1*上的為沒有仲裁見證使用簡單的節點多數設定。

```PowerShell
Set-ClusterQuorum –Cluster CONTOSO-FC1 -NodeMajority
```

下列範例會變更在本機叢集仲裁設定為節點大部分與見證設定。 名為*叢集磁碟 2*的磁碟資源會設定為磁碟見證。

```PowerShell
Set-ClusterQuorum -NodeAndDiskMajority "Cluster Disk 2"
```

下列範例會變更在本機叢集仲裁設定為節點大部分與見證設定。 名為*\\\CONTOSO-FS\\fsw*檔案共用資源會設定為檔案共用見證。

```PowerShell
Set-ClusterQuorum -NodeAndFileShareMajority "\\fileserver\fsw"
```

下列範例會從本機叢集的節點*ContosoFCNode1*移除仲裁投票。

```PowerShell
(Get-ClusterNode ContosoFCNode1).NodeWeight=0
```

下列範例會新增到本機叢集上的節點*ContosoFCNode1*仲裁投票。

```PowerShell
(Get-ClusterNode ContosoFCNode1).NodeWeight=1
```

下列範例 （如果它先前停用），可讓叢集*CONTOSO FC1* **DynamicQuorum**屬性：

```PowerShell
(Get-Cluster CONTOSO-FC1).DynamicQuorum=1
```

## 沒有仲裁啟動修復叢集

沒有足夠的仲裁投票在叢集中不會啟動。 第一個步驟是，您應該一律確認叢集仲裁設定，並調查為何不再有仲裁。 如果您有節點的停止回應，或如果主要站台不是在多站台叢集中保持連線，可能會發生此情況。 您找出根本原因，叢集失敗後，您可以使用本節中所述的修復步驟。

>[!NOTE]
> * 如果叢集服務停止，因為仲裁會遺失，事件識別碼 1177年會出現在系統記錄檔中。
> * 總是必要調查為什麼叢集仲裁已遺失。
> * 一律最好是將帶而不從沒有仲裁叢集節點或仲裁見證良好的狀態 （加入叢集）。

### 強制 [開始] 畫面叢集節點

您判斷您無法透過將節點或仲裁見證，良好的狀態復原您的叢集之後，就需要強制啟動您的叢集。 強制以開始叢集會覆寫您叢集中的仲裁組態設定，並開始**ForceQuorum**模式中的叢集。

強制它沒有仲裁時要啟動的叢集可能會在多站台叢集中特別有用。 請考慮包含分別位於主要和備份的網站， *SiteA*和*SiteB*在叢集中的嚴重損壞修復案例。 如果在*SiteA*正版發生嚴重損壞，可能需要很長的時間來重新上線的網站。 您可能會想要強制*SiteB*上線，即使它沒有仲裁。

在叢集中**ForceQuorum**模式中，在啟動時，而且之後它重新取得足夠仲裁投票、 叢集自動離開的強制的狀態，並正常的行為。 因此，不需要再次正常啟動叢集。 如果叢集失去節點，以及它失去仲裁，它會離線再次因為不再的強制狀態。 若要使其回復上線時沒有仲裁需要強制啟動，而仲裁叢集。

>[!IMPORTANT]
>* 叢集是強制啟動之後，系統管理員就是在叢集的完整控制權。
>* 叢集叢集所在動力開始，在節點上使用叢集設定，並將它複寫到所有其他可用的節點。
>* 如果您強制啟動，而仲裁叢集，叢集保持在**ForceQuorum**模式時，會忽略所有仲裁組態設定。 這包括特定節點投票指派和動態仲裁管理設定。

### 防止剩餘的叢集節點上的仲裁

強制之後開始叢集節點上，是為了在您叢集上設定，以防仲裁使用開始任何剩餘的節點。 開始使用，以防止仲裁設定的節點表示要加入現有的執行叢集，而不是以形成新的叢集執行個體的叢集服務。 如此可防止的其餘節點以形成分割叢集，其中包含兩個激烈的執行個體。

這就變得必要時您需要之後有動力復原您的叢集，在某些多站台發生嚴重損壞修復案例中您備份在網站上， *SiteB*啟動叢集。 要加入強制啟動叢集*SiteB*，在您的主要站台， *SiteA*節點中，需要開始禁止的仲裁使用。

>[!IMPORTANT]
>叢集節點上開始使用強制之後，我們建議您一律禁止的仲裁使用啟動的其餘節點。

以下是如何修復叢集使用容錯移轉叢集管理員：

1. 在容錯移轉叢集管理員中，選取，或指定您想要修復的叢集。
2. 與叢集**動作**下, 選取，選取 [**強制叢集開始**。

    容錯移轉叢集管理員動力連線到的所有節點上啟動叢集。 開始時的叢集會使用目前的叢集設定。

>[!NOTE]
>* 若要強制以開始上特定的節點，其中包含您想要使用叢集設定叢集，您必須使用 Windows PowerShell cmdlet 或對等的命令列工具，此程序之後呈現。 
> * 如果您使用容錯移轉叢集管理員來連線到叢集是強制開始，且您使用**啟動叢集服務**動作來啟動節點，節點會自動啟動，並設定，以防止仲裁。

#### Windows PowerShell 對等的命令 (Clusternode 開始)

下列範例示範如何使用 **[開始] 畫面 ClusterNode** cmdlet 強制 [開始] 畫面叢集節點*ContosoFCNode1*上。

```PowerShell
Start-ClusterNode –Node ContosoFCNode1 –FQ
```

或者，您可以在節點上本機輸入下列命令：

```PowerShell
Net Start ClusSvc /FQ
```

下列範例示範如何使用 **[開始] 畫面 ClusterNode** cmdlet 以防止節點*ContosoFCNode1*上發生的仲裁開始叢集服務。

```PowerShell
Start-ClusterNode –Node ContosoFCNode1 –PQ
```

或者，您可以在節點上本機輸入下列命令：

```PowerShell
Net Start ClusSvc /PQ
```

## 發生嚴重損壞修復組態的仲裁考量

本節摘要說明特性和兩個多站台的叢集設定，在發生嚴重損壞修復部署的仲裁設定。 如果您需要進行自動容錯移轉或手動容錯移轉為站台之間的工作負載取決於與不同的仲裁設定的指導方針。 您的設定通常取決於服務等級協定 (Sla) 中提供，並支援發生失敗或在站台發生嚴重損壞時的叢集工作負載在組織中的位置。
