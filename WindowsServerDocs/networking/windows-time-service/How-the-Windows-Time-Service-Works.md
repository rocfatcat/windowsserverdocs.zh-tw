Windows 時間服務不精確實作網路時間通訊協定 (NTP)，但它將會使用確保為越準確時鐘電腦在網路上的 NTP 規格中所定義的套件複雜的演算法。 最好 AD DS 網域中的所有電腦時鐘會同步都處理的授權電腦的時間。 許多因素可能會影響網路上的同步處理時間。 下列因素通常會影響到 AD DS 正在同步之正確性：  
  
-   網路條件  
  
-   電腦硬體時鐘之正確性  
  
-   數量的 CPU 及網路資源可供 Windows 時間服務  
  
> [!IMPORTANT]  
> Windows Server 2016 之前 W32Time 服務設計時效性的應用程式需求。  不過，Windows Server 2016 的更新現在可讓您實作的 1ms 方案網域中的準確度。  查看[Windows 2016 正確時間](accurate-time.md)和[設定為高不正確的環境 Windows 時間服務支援邊界](https://go.microsoft.com/fwlink/?LinkID=179459)的詳細資訊。  
  
根據預設，若要同步的 time.windows.com 經常同步他們的使用時間較或未加入網域的電腦的設定。因此，很難保證電腦上，有暫時性或網路連接時間準確度。  
  
AD DS 森林有同步階層預先定義的時間。 Windows 時間服務會同步處理中階層、 上方的最精確參考時鐘與電腦之間的時間。 如果您的電腦上設定一個以上的時間來源，Windows 的時間會 NTP 演算法從依據電腦的功能與的時間來源同步的設定來源選取最佳的時間來源。 Windows 時間服務不支援網路廣播 」 或 「 多點對等裝置的同步處理。 如需有關這些 NTP 功能的詳細資訊，查看 RFC 1305 IETF RFC 資料庫中。  
  
每一部電腦正在執行 Windows 時間服務使用服務以維護最正確的時間。 預設成員網域的電腦做為時間 client，因此，通常不需要設定「Windows 時間服務。 不過，Windows 時間服務時間要求來自參考指定的時間來源，您可以設定，也可以用戶端提供時間。
  
電腦的時間是正確的角度稱為階層。 （例如硬體時鐘） 網路上的最正確的時間來源占的最低組織層級或組織層一個。 此正確的時間來源稱為參考時鐘。 取得參考時鐘直接從時間 NTP 伺服器占階層高於參考時鐘一個層級。 資源，取得 NTP 伺服器的時間的兩個步驟，參考時鐘]，因此佔據組織層的兩個高於最正確的時間來源，並等等。 為電腦的組織層的數字增加，其系統時鐘時間可能會變得較不精確。 因此的任何電腦組織層級是指示器與最正確的時間來源程度同步該電腦。  
  
當 W32Time 管理員收到時間範例時，它會使用特殊的演算法 NTP 中判斷哪一次範例是適用於。 時間服務也會使用另一組演算法判斷是最精確的其中一項設定的時間來源。 時的時間範例是最好的作法發現時間服務，讓它在正確時間整合的本機時鐘速率根據上述條件，調整。 如果間的本機時鐘與 （也稱為與的時間偏差） 選取正確的時間範例更正調整本機時鐘速度太大，時間服務會將本機時鐘為正確的時間。 這項調整時鐘速率或直接時鐘時間變更稱為時鐘規範。

## <a name="w2k3tr_times_how_ekoc"></a>Windows 時間服務時間通訊協定  

時間通訊協定判斷程度兩部電腦正在時鐘。 判斷的最佳使用時間資訊和影時鐘來確保一致的時間會在不同的系統上維護時間通訊協定負責。  
  
Windows 時間服務使用網路時間通訊協定 (NTP)，以協助您在網路上同步處理時間。 NTP 是包含訓練演算法所需的同步時鐘網際網路時間通訊協定。 NTP 是更加準確的時間通訊協定比簡單網路時間通訊協定 (SNTP) 用於某些版本的 Windows。不過 W32Time 會繼續支援 SNTP，可讓電腦執行的時間 SNTP 為基礎的服務，例如 Windows 2000 的回溯相容性。  
  
### <a name="network-time-protocol"></a>網路時間通訊協定  
預設值為網路時間通訊協定 (NTP) 作業系統 Windows 時間服務使用同步處理通訊協定的時間。 NTP 容錯、 高度延展性時間通訊協定，且使用指定的時間的參考同步電腦時鐘最常使用的通訊協定。  
  
同步處理時間 NTP 發生一段時間，而在網路上包括 NTP 封包轉換。 NTP 封包包含時間戳記同步處理時間中包含 client 和參與伺服器的時間範例。  
  
NTP 依賴定義最正確的時間，使用參考時鐘與同步該參考時鐘網路上的所有時鐘。 NTP 使用目前的時間通用標準國際標準時間 (UTC)。 UTC 不受影響的時區，並讓 NTP 無論時區設定世界任何位置使用。  
  
#### <a name="ntp-algorithms"></a>NTP 演算法  
NTP 包含兩種演算法、 時鐘篩選演算法和時鐘選擇演算法，以協助判斷最佳的時間範例 Windows 時間服務。 時鐘篩選演算法的設計目的是篩選會接收查詢的時間來源，而且判斷最佳時間範例每個來源的時間範例。 時鐘選擇演算法然後判斷最精確網路上的時間伺服器。 這項資訊是然後傳遞至時鐘訓練演算法，使用時補償網路延遲和電腦時鐘留意因為錯誤修正本機電腦的時鐘所收集的資訊。  
  
NTP 演算法就是最精確的燈號-中度網路及伺服器負載的條件。 與網路的傳輸時間會考量任何演算法，為 NTP 演算法可能不良條件極端網路壅塞在執行。 如需 NTP 演算法，查看 RFC 1305 IETF RFC 資料庫中。  
  
#### <a name="ntp-time-provider"></a>NTP 時間提供者  
Windows 時間服務是完整時間同步套件，可支援各種不同的硬體裝置與的時間通訊協定。 若要使用這項支援，請服務會使用可插入時間提供者。 時間提供者負責任一取得正確的時間戳記 （從網路或硬體） 的或提供其他電腦的時間戳記在網路上。  
  
NTP 提供者已隨附作業系統標準時間提供者。 NTP 提供者遵守指定 NTP client 和 server 第 3 版標準，可以互動 SNTP 戶端和 Windows 2000 與其他 SNTP 戶端回溯相容性的伺服器。 Windows 時間服務提供者 NTP 下列兩個組件包含：  
  
-   **NtpServer 輸出提供者。** 這是回應 client 時間要求網路上時間伺服器。  
  
-   **NtpClient 輸入提供者。** 這是時間 client 取得時間資訊從其他來源硬體裝置或 NTP 伺服器，並可以退還，很適合同步處理本機電腦時鐘時間範例。  
  
雖然密切相關的實際下列兩個提供者作業，它們會出現獨立時間服務。 從 Windows 2000 Server、 Windows 的電腦已連接到網路，它被設定為 NTP client。 此外，電腦執行的 Windows 時間服務嘗試只預設同步處理時間網域控制站或手動指定的時間來源。 這些是時間的慣用的時間提供者，因為它們是時間的使用自動、 安全來源。  
  
#### <a name="ntp-security"></a>NTP 安全性  

AD DS 樹系中 Windows 時間服務依賴執行的時間資料驗證標準網域的安全性功能。 做為時間伺服器的本機網域控制站成員網域的電腦之間傳送 NTP 封包的安全性根據共用金鑰驗證。 Windows 時間服務會使用電腦的 Kerberos 工作階段金鑰來建立 NTP 封包傳送在網路上的已驗證的特徵標記。 不在安全網路登入通道傳輸 NTP 封包。 改當電腦從網域階層網域控制站要求與的時間，Windows 時間服務會要求驗證與的時間。 網域控制站的形式 64 位元值工作階段金鑰從網路登入服務已經過驗證，再傳回所需的資訊。 如果傳回的 NTP 封包使用電腦工作階段金鑰未經簽署或已不正確，遭拒與的時間。 所有這類驗證失敗的登入事件登入。 如此一來，Windows 時間服務會提供 AD DS 森林中 NTP 資料安全性。  
  
一般而言，Windows 時間戶端自動同步處理的正確時間從取得網域控制站在相同的網域。 樹系子女網域中的網域控制站與他們家長網域中的網域控制站同步處理時間。 當時間伺服器回到要求時間 client 的已驗證的 NTP 封包時，Kerberos 工作階段按鍵跨網域信任 account 所定義透過已封包。 跨網域信任 account 建立新 AD DS 網域結合樹系時網路登入服務管理工作階段金鑰。 如此一來，設定為可靠森林根網域中的網域控制站在所有的網域控制站父系和子女在網域中，以及間接所有的電腦位於網域樹系的已驗證的時間來源。  
  
Windows 時間服務可以之間森林，才能設定，但請務必注意，此設定不安全。 例如，NTP 伺服器可能可以在不同的樹系。 不過，因為該電腦以不同的樹系，不是 Kerberos 工作階段金鑰，用來登入和驗證 NTP 封包。 在不同的樹系的電腦取得正確的時間同步，client 需要的網路存取權的電腦，必須設定時間服務使用其他樹系中的特定時間來源。 如果從 NTP 伺服器以外自己網域階層手動 client 設定為存取時間，傳送 client 與的時間伺服器的 NTP 封包未驗證，因此不安全。 信任的樹系實作，即使有 Windows 時間服務不安全跨樹系。 雖然安全網路登入通道驗證機制 Windows 時間服務，不支援跨樹系驗證。  
  
#### <a name="hardware-devices-that-are-supported-by-the-windows-time-service"></a>Windows 時間服務所支援的硬體裝置  
硬體式的時鐘，例如 GPS 或廣播時鐘常用高準確參考時鐘裝置。 根據預設，直接連接到電腦的硬體裝置不支援 Windows 時間服務 NTP 時間提供，雖然您可以建立軟體獨立時間提供者的支援這種類型的連接。 這種類型的搭配 Windows 時間服務提供者可以提供可靠，穩定時間參考。  
  
硬體裝置，例如時鐘 cesium 或全球定位系統 (GPS) 接收器，提供下列以取得正確的時間定義標準準確目前的時間。 Cesium 時鐘是非常穩定，並不會受到因素溫度、 壓力或濕度，例如但也非常高。 GPS 接收器是更便宜操作，也準確參考時鐘。 GPS 接收器從 cesium 時鐘取得他們的使用時間的衛星取得他們的使用時間。 獨立時間提供使用，而 Windows 的時間伺服器可以連接到外部 NTP 伺服器，這透過電話或網際網路連接到硬體裝置取得他們的使用時間。 例如美國海 Observatory 組織提供 NTP 伺服器非常可靠參考時鐘連接。  
  
許多 GPS 接收器的其他裝置的時間可以作為 NTP 網路上的伺服器。 您可以設定 AD DS 樹系才也擔任 NTP 伺服器，您網路上同步處理這些外部硬體裝置的時間。 若要這樣做，請設定的網域控制站的主要網域控制站運作 （） 肯定在您的樹系根與 GPS 裝置提供 NTP 伺服器同步。 若要這樣做，請查看[上的樹系根網域中肯定設定 Windows 時間服務](https://docs.microsoft.com/en-us/previous-versions/windows/it-pro/windows-server-2008-R2-and-2008/cc731191%28v=ws.10%29)。  
  
### <a name="simple-network-time-protocol"></a>簡單網路時間通訊協定  
簡單網路時間通訊協定 (SNTP) 是正確性的旨在為伺服器及不需要 NTP 提供程度簡化的時間通訊協定。 SNTP，更基本 NTP 的版本是使用 Windows 2000 的主要時間通訊協定。 由於 SNTP 和 NTP 網路封包格式相同的兩個通訊協定而且交互作用。 這兩個主要不同是 SNTP 未錯誤管理及複雜篩選系統 NTP 提供。 如需簡單網路時間通訊協定，查看 RFC 1769 IETF RFC 資料庫中。  
  
### <a name="time-protocol-interoperability"></a>時間通訊協定交互操作  
Windows 時間服務可以在的電腦執行的 Windows 2000、 Windows XP 和 Windows Server 2003，混合的環境中運作，因為使用在 Windows 2000 的 SNTP 通訊協定交互的 Windows XP 和 Windows Server 2003 NTP 通訊協定作用。  
  
Windows Server nt4.0，稱為 「 TimeServ，在服務時間同步 Windows nt4.0 網路上的時間。 TimeServ 是附加元件功能提供的一部分*Microsoft NT 4.0 資源套件*，並不提供程度同步處理時間所需的 Windows Server 2003 的可靠性。  
  
Windows 時間服務可以交互的電腦是執行 Windows nt4.0，因為它們可以與 Windows 2000 或 Windows Server 2003; 電腦同步處理時間不過，執行 Windows 2000 或 Windows Server 2003 的電腦不會不會自動探索 Windows nt4.0 時間伺服器。 例如如果您的網域設定為使用階層為基礎的網域同步處理時間想來與 Windows nt4.0 網域控制站同步處理時間在的電腦同步處理和您的方法，您必須設定這些電腦以手動方式與 Windows nt4.0 網域控制站同步。  
  
Windows nt4.0 比 Windows 時間服務使用同步處理時間使用的簡易機制。 因此，以確保您網路上的正確時間同步處理，建議您升級到 Windows 2000 或 Windows Server 2003 任何 Windows nt4.0 網域控制站。  
  
## <a name="w2k3tr_times_how_izcr"></a>Windows 時間服務程序與互動  

Windows 時間服務的設計可以同步的網路上的電腦。 網路時間同步處理程序，也稱為時間聚合，就會發生在網路為電腦存取每當更加準確的時間伺服器。 時間聚合包括授權伺服器 NTP 封包的形式 client 電腦目前的時間提供處理程序。 封包中所提供的資訊，表示對目前在電腦的時間，讓它更加準確伺服器與同步是否需要調整。  
  
時間聚合程序的一部分，嘗試使用相同的網域中的任何網域控制站同步處理時間網域成員。 如果電腦位於網域控制站，它會嘗試使用更多可靠的網域控制站同步處理。  
  
電腦執行的是 Windows XP Home Edition 或未加入網域的電腦不嘗試同步的網域階層、，但是從 time.windows.com 取得時間預設設定。  
  
若要建立的電腦執行 Windows Server 2003 授權，電腦必須設定為可靠的時間來源。 根據預設，第一次網域控制站的 Windows Server 2003 網域上已安裝會自動設定為可靠的時間來源。 因為授權網域的電腦，它必須外部的時間來源，而網域階層同步設定。 也根據預設，所有其他成員的 Windows Server 2003 網域設定的網域階層同步。  
  
Windows Server 2003 網路建立後，您可以設定 Windows 時間服務使用同步下列選項之一：  
  
-   網域型階層同步  
  
-   手動指定同步來源  
  
-   所有可用的同步處理機制  
  
-   不是同步。  
  
每個同步類型討論下一節。

#### <a name="reliable-time-source-configuration"></a>可靠的時間來源的設定  
電腦設定為可靠的時間來源被視為根的時間服務。 根時間服務的網域授權伺服器，通常會設定為從硬體裝置或外接 NTP 伺服器擷取的時間。 為可靠的時間來源最佳化網域階層整個傳送時間的方式可以設定時間伺服器。 如果網域控制站設定為可靠的時間來源，網路登入服務宣告網域控制站為可靠的時間來源登入時網路。 其他網域控制站的時間來源，若要同步的外觀時, 選擇可靠的來源第一次如果有的話。  
  
#### <a name="time-source-selection"></a>時間來源選取項目  
時間來源選擇程序可以建立網路上的兩個問題：  
  
-   同步處理其他循環。  
  
-   網路流量提高的音量。  
  
在 [同步網路循環發生於時間維持一致之間的網域控制站群組，並在這些持續不與其他可靠的時間來源同步之間共用的同時。 Windows 時間服務的時間來源選擇演算法被設計來抵禦這些類型的問題。  
  
電腦使用其中一項下列方法找出要同步的時間來源：  
  
-   如果您的電腦不是網域中的成員，它必須與指定的時間來源同步設定。  
  
-   如果電腦是成員伺服器或工作站預設網域中的遵循 AD DS 階層並與目前執行的是 Windows 時間服務其本機網域中的網域控制站同步處理時間。  
  
如果電腦位於網域控制站，這會讓尋找同步處理的其他網域控制站最多六個查詢。 找出的某些屬性，例如網域控制站的特定位置，一種的時間來源的設計目的是每個查詢且是否可靠的時間來源。 時間來源必須也遵守下列限制：  
  
-   家長網域中的網域控制站只可以同步可靠的時間來源。  
  
-   它自己網域中的可靠的時間來源或家長網域中的任何網域控制站可以同步肯定。  
  
如果網域控制站無法同步的網域控制站它是查詢類型，不會對查詢。 網域控制站知道的電腦的時間才能取得才查詢哪一種。 例如，本機肯定不會嘗試查詢數字三或六個因為網域控制站不會與本身同步。

### <a name="manually-specified-synchronization"></a>手動指定同步  
手動指定同步可讓您指定單一對等裝置或從的電腦取得時間對等裝置清單。 如果您的電腦不是網域中的成員，它必須手動設定與指定的時間來源同步。 電腦位於網域同步網域階層預設設定，以手動方式指定同步是實用的樹系的網域根或適用於未加入網域的電腦。 手動指定外部 NTP 伺服器與您的網域中的授權電腦同步處理提供可靠的時間。 不過，設定授權硬體時鐘與同步您的網域的電腦是實際好方案提供您的網域最精確、 安全時間。  
  
手動指定的時間來源是不會經過寫入特定時間提供者，以及因此容易受到攻擊。 同時，如果您的電腦同步處理手動指定的來源，而非的驗證網域控制站，兩部電腦可能不同步，導致 F:kerberos 驗證失敗。 這可能會要求網路驗證失敗，例如列印或分享檔案的其他動作。 如果只有樹系根同步外部來源的設定，在森林中的所有其他電腦保持同步合作，讓難以重新執行攻擊。  
  
### <a name="all-available-synchronization-mechanisms"></a>所有可用的同步處理機制  

「 所有可用的同步處理機制 」 是最有價值的使用者，在網路上同步處理方式。 這個方法可同步的網域階層，也可能提供其他網域階層無法使用，而定設定的時間來源。 如果 client 無法同步網域階層與的時間的時間來源自動退回所指定的時間來源**NtpServer**設定。 同步處理的方法是正確的時間提供給戶端最有可能。  

### <a name="stopping-time-synchronization"></a>停止同步處理時間  
有的著中您會想要停止同步時間的電腦。 例如，電腦嘗試透過撥號透過 WAN 同步從網際網路上的時間來源或其他網站時，如果它可以收取高電話費用。 停用該電腦上的同步處理之後，您避免電腦嘗試存取的時間來源透過撥號。  
  
您也可以同步，以避免錯誤事件登入代停用。 每次的電腦嘗試同步與的時間來源，無法使用，則會在事件登入產生錯誤。 如果您的時間來源使用網路上的維護引導您不想要重新 client 同步從其他來源的設定，您可以停用防止進行同步處理時間伺服器時，無法使用 client 上的同步處理。  
  
很有幫助停用指定為根同步處理網路的電腦上同步處理。 這表示電腦根信任本機時鐘。 如果根同步階層的不設定為**非同步**，如果您無法同步處理的其他的時間來源，戶端不接受這台電腦送出，因為它的時間不會受信任的封包。  
  
只伺服器，即使它們不同步處理的其他的時間來源用受信任的這些已被視為 client 的可靠的時間伺服器。  
  
### <a name="disabling-the-windows-time-service"></a>停用 Windows 時間服務  
Windows 時間服務 (W32Time) 可以完全停用。 如果您選擇實作使用 NTP 第三方時間同步 product，您必須停用 Windows 時間服務。 這是因為所有 NTP 伺服器都需要存取權的使用者資料流通訊協定 (UDP) 連接埠 123，並連接埠 123，只要在 Windows Server 2003 的作業系統上執行的 Windows 時間服務，仍時間 Windows 所保留。
