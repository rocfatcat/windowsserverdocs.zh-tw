# <a name="server-performance-advisor-users-guide"></a>Server Performance Advisor 使用者指南

>適用于： Windows Server (半年通道) 、Windows Server 2016、Windows Server 2012 R2、Windows Server 2012、Windows 10 Windows 8

本 Microsoft Server Performance Advisor 的使用者指南 (SPA) 提供有關如何使用 SPA 來識別部署于各種伺服器角色之系統效能瓶頸的指導方針。

SPA 可協助您進行下列作業：

* 管理您的伺服器效能，並針對伺服器效能問題進行疑難排解。

* 提供有關常見設定和效能問題的資料包表和建議。

* 根據所收集的資料，提供最佳做法建議。

> [!NOTE]
> SPA 主控台不會對伺服器進行任何變更。

如需開發 SPA Advisor 套件的詳細資訊，請參閱 [伺服器效能建議程式套件開發指南](server-performance-advisor-pack-development-guide.md)。

## <a name="server-performance-advisor-overview"></a>伺服器效能審查程式總覽


本節說明 SPA 的歷程記錄、描述其目標物件和案例，並提供工具的架構總覽。

### <a name="history-of-spa"></a>SPA 的歷程記錄

Microsoft Server Performance Advisor (SPA) 的原始版本已于2005-2006 發行。 它主要是以 Windows Server 2003 為目標，包括核心作業系統和伺服器角色，例如 Internet Information Services (IIS) 和 active directory。 這項工具的目標是協助您評估伺服器效能，以及針對伺服器效能問題進行疑難排解。

SPA 會為系統管理員提供有關常見設定和效能問題的資料包告和建議。 SPA 會收集伺服器上各種來源的效能相關資料，例如效能計數器、登錄機碼、WMI 查詢、設定檔，以及 Windows (ETW) 的事件追蹤。 SPA 以其所收集的伺服器效能資料為基礎，可提供目前伺服器效能狀況的深入探討，並提出有關可改進之事項的建議。

原始的 SPA 工具有超過1000000的下載，它説明許多系統管理員管理伺服器的效能。 它是一項成功的效能疑難排解工具。

從原始 SPA 的版本開始，伺服器效能世界中有幾項重大變更。 最值得注意的是，Windows Server 2012 R2、Windows Server 2012、Windows Server 2008 R2 和 Windows Server 2008 的發行版本，有新版本的對應伺服器角色，例如 Internet Information Services (IIS) 8.5。 較新版本的伺服器效能建議程式會將原始 SPA 的功能延伸至最新的伺服器作業系統和伺服器角色。

雖然 SPA 3.1 與原始工具共用相同的設計目標和效能分析範例，但它已經以最新的技術進行重寫。 它也有下列重要的改進：

* 可以針對執行 Windows Server 2012 R2、Windows Server 2012、Windows Server 2008 R2 和 Windows Server 2008 的伺服器執行分析。

* 支援遠端分析功能，可讓 SPA 3.1 從中央主控台收集和分析資料，而不需要在要分析的伺服器上安裝程式碼。

* 使用資料分析和儲存的 Microsoft SQL Server，可讓您處理和儲存大量資料。

* 分隔工具與 Advisor 套件。 每個伺服器角色的效能分析邏輯都是以 advisor 套件的形式發行，可與工具分開發行或更新。

* 提供在 SQL 腳本中使用開放式架構開發的 Advisor 套件。 非 Microsoft 的當事人可開發 advisor 套件或擴充 Microsoft 現有的 advisor 套件，以涵蓋特殊需求。

* 支援並存比較報告等新功能，以及趨勢和歷程記錄圖表，以協助您找到異常狀況。

* 提供修改、匯入和匯出臨界值之類的功能，可協助您微調報表和通知，並與其他 SPA 使用者共用這些並。

* 支援多個專案，可用來將目標伺服器分組。

* 從 SPA 主控台內提供週期性的資料收集。

* 使用) 的 advanced users Microsoft SQL Server (，啟用自訂查詢和報告產生。

* 自訂的 Windows PowerShell Cmdlet 可搭配 SPA 使用

* 從 SPA 3.0 資料庫匯入和查看報表的回溯相容性

### <a name="target-audience"></a>目標對象

SPA 工具主要是針對管理各種伺服器角色中少於100部伺服器的系統管理員所設計。 支援工程師也可以使用它來收集效能資料，以及針對客戶的效能問題進行疑難排解。

SPA 提供的建議可協助您解決效能問題;不過，它是根據可能不適用於您的特定伺服器環境的假設。 透過 SPA 提供的建議應視為建議。 我們預期 SPA 使用者對系統組態、使用案例，以及系統調整對整體系統行為的影響，都有充分的瞭解。 系統管理員應決定 SPA 建議是否應套用至其環境。

### <a name="target-scenarios"></a>目標案例

以下是 SPA 的目標案例：

* **伺服器環境**

    SPA 是設計用來輕鬆設定和維護。 它適用于使用1到100伺服器的伺服器環境。 如果您嘗試管理100部以上的伺服器效能，SPA 將無法妥善調整。 針對較大型或更複雜的環境，您應該考慮使用 System Center Operations Manager。

* **效能疑難排解**

    您可以使用 SPA 作為效能疑難排解工具。 它提供收集高階效能資料的功能，它會對資料執行完整的 post 處理，讓您更瞭解整體系統行為，並將任何異常標記為任何異常。 當客戶懷疑有效能問題時，您可以使用 SPA 來收集和分析伺服器的效能資料。

    SPA 會產生可供您查看的報表。 SPA 報表提供可醒目提示潛在問題的通知清單，以及包含伺服器之各種效能索引、設定和設定的資料區段。 您可以使用此報表來識別特定效能問題，並使用建議來尋找問題的解決方案。 報表可以與其他在不同時間或不同伺服器上產生的報表進行比較。 您可以使用這個並存比較來判斷基準一般與異常行為之間的差異。

    **注意** SPA 並非設計成可作為偵錯工具或計量工具。 此外，從伺服器的觀點來看，它可以被視為唯讀的工具，而不會修改伺服器設定。



* **效能索引監視**

    您可以使用 SPA 來監視伺服器的效能索引。 您可以選擇在伺服器上定期執行 SPA 來收集效能資料，然後執行趨勢圖或歷程記錄圖表以找出異常狀況。 您可以查看特定分析的報表、找出效能問題的詳細資訊，然後使用建議或其他報表資料來解決問題。

### <a name="spa-architecture"></a>SPA 架構

SPA 資料收集邏輯是以 Windows 中的通訊協定為基礎，稱為效能記錄檔及警示 (PLA) 。 PLA 可讓程式收集本機或遠端伺服器的效能資料，例如效能計數器、WMI 查詢、ETW 追蹤、登錄機碼和設定檔案。 當 SPA 在目標伺服器上執行效能分析時，它會根據您選取的特定 SPA 建議程式套件來建立 PLA 資料收集器集合。 Advisor 套件包含要收集的資料來源，以及要儲存記錄檔的檔案共用。 SPA 資料收集只會儲存單一使用者帳戶;您也可以使用 PLA 所使用的相同使用者帳戶來寫入記錄。

SPA 使用 SQL Server 資料庫來儲存從目標伺服器收集而來的效能記錄。 SPA 會將檔案共用中的所有效能記錄檔匯入到資料庫，然後使用每個 advisor 套件內的資料分析邏輯來處理資料和產生報告。 Advisor 套件會分析從目標伺服器收集的效能資料，並產生 SPA 報告。 如需建立 advisor 套件的詳細資訊，請參閱 [伺服器效能建議程式套件開發指南](server-performance-advisor-pack-development-guide.md)。

SPA 主控台使用者介面和互動會建立為 SPAConsole.exe 的一部分。 您可以使用主控台來建立資料庫、新增或移除 advisor 套件、管理目標伺服器、執行效能分析，以及查看效能報告。

SPA 主控台可在下列作業系統上執行：

* Windows 8.1

* Windows 8

* Windows 7

* Windows Server 2012 R2

* Windows Server 2012

* Windows Server 2008 R2

* Windows Server 2008

在一般商務應用程式中，有三個層級：展示層、商務邏輯層和儲存層。 SPA 是設計為主控台和資料庫的兩層產品。 主控台可作為包含特定處理常式相關邏輯的展示層，且資料庫可作為儲存層和商務邏輯層。 主控台會捕獲使用者輸入，並控制資料收集、資料處理和報告產生的步驟。 SPA 不相依于 Windows 系統服務。

下圖顯示 SPA 系統的高層級架構。 程序如下：

1.  從 SPA 主控台，您可以在指定的伺服器上執行效能分析。

2.  收集效能資料時，目標伺服器上的 PLA 會將記錄檔寫回資料收集器集合人員所指定的檔案共用。

3.  在目的電腦上完成資料收集之後，SPA 主控台會將記錄檔匯入 SQL Server 資料庫。

4.  主控台會叫用特定 advisor 套件的資料處理邏輯。

5.  Advisor 套件會處理資料，並呼叫 SPA Api，以將記錄插入 SPA 架構所定義的系統報告資料表中。

6.  您可以使用主控台內的報表檢視器來查看產生的報告。 為了協助您針對效能問題進行疑難排解，SPA 提供三種類型的報告：單一報表、並存報表，以及趨勢或歷程記錄圖表。 如需報表的詳細資訊，請參閱 [查看報表](#bkmk-viewingreports)。

## <a name="getting-started-with-spa"></a>SPA 入門


### <a name="requirements"></a>需求

SPA 主控台可安裝在 Windows 8.1、Windows 8、Windows 7、Windows Server 2012 R2、Windows Server 2012、Windows Server 2008 R2 和 Windows Server 2008。 不支援在舊版 Windows Server 作業系統上執行 SPA。 SPA 是在 x86 或 x64 上執行，但不支援 IA64 或 ARM 架構。

SPA 建基於 Windows Presentation Foundation (WPF) 2.0，這是 Microsoft .NET Framework 4 的一部分，因此需要 .NET Framework 4。

您也需要在安裝 SPA 的相同電腦上安裝 SQL Server 2008 R2 Express。 SQL Server 2008 R2 Express 是 Microsoft 所發行的免費資料庫引擎。 您可以從 Microsoft 下載中心下載 Microsoft SQL Server 2008 R2 Express。 雖然我們建議 SQL Server 2008 R2 Express，但較新版本的 SQL Server 可能也與 SPA 相容。

**注意** SPA 不包含 SQL Server 或 .NET Framework 作為 SPA 安裝套件的一部分。 安裝 Microsoft SQL Server 2008 R2 和 .NET Framework 4.0 之後，建議您在安裝 SPA 之前先執行 Windows Update。



因為使用者可以建立及管理具有 SPA 的資料庫，所以用來執行 SPA 的使用者帳戶應該具有與 SQL Server 相同的系統管理員許可權。

### <a name="setting-up-spa"></a><a href="" id="bkmk-setupspa"></a>設定 SPA

SPA 封裝為 .cab 檔案，其中包含 SPA 架構的所有二進位檔、用於 advanced 案例的 Windows PowerShell Cmdlet，以及下列 advisor 套件：核心 OS、Hyper-v、active directory 和 IIS。 將 .cab 檔解壓縮到資料夾之後，就不需要進行任何額外的安裝。 不過，若要執行 SPA，您需要從目標伺服器啟用資料收集，如下所示：

* 若要執行 PLA 資料收集，您用來執行 SPA 主控台的使用者帳戶必須是目標伺服器上 Administrators 安全性群組的一部分。 如果目標伺服器和主控台都在相同的網域中，則網域使用者帳戶必須是目標伺服器上 [系統管理員] 安全性群組的一部分。 如果目標伺服器和主控台不在相同的網域中，請在目標伺服器上建立系統管理使用者帳戶，其使用者名稱和密碼與您用來執行 SPA 主控台的使用者帳戶相同。

* 在伺服器上建立結果的共用資料夾。

* 請確定您用來執行 SPA 主控台的使用者帳戶具有共用資料夾的讀取和寫入權限。 PLA 會使用此帳戶將記錄寫入資料夾。
SPA 主控台會使用相同的帳戶來讀取記錄，並將它們匯入資料庫中。

    **注意** ETW 會執行迴圈緩衝區來儲存追蹤，並盡可能將它們移至共用資料夾。 如果伺服器忙碌中或寫入作業緩慢，ETW 會在緩衝區已滿時卸載追蹤。 共用資料夾必須位於能夠快速存取 i/o 的伺服器上。 我們建議每部目標伺服器都有共用資料夾，以將檔案 i/o 緩慢造成的資料遺失降至最低。



* 若要讓 PLA 存取目標伺服器，請將 Windows 防火牆設定為允許在目標伺服器上進行遠端效能記錄檔及警示存取。 PLA 使用 TCP 埠139。

* 請確認 **效能記錄 & 警示** 服務正在執行中。

* 如果主控台和目標伺服器位於不同的子網，您也需要在 [**效能記錄檔及警示**] 頁面的 [**範圍**設定] 的 [輸入防火牆規則] 中設定 [遠端 IP 位址] 欄位，如下所示。

    ![pla 屬性](../media/server-performance-advisor/spa-user-manual-pla-firewall.png)

* 開啟主控台和每部目標伺服器上的網路探索。

* 如果目標伺服器未加入網域，請啟用下列登錄設定： **HKLM \\ SOFTWARE \\ Microsoft \\ Windows \\ Currentversion \\ 原則 \\ 系統 \\ LocalAccountTokenFilterPolicy**。

**注意** 根據預設，SPA 會將診斷記錄寫入 SpaConsole.exe 所在的資料夾。 如果 SPA 安裝在 [Program Files] 資料夾下，只有在以系統管理員身分執行 SpaConsole.exe 時，SPA 才能夠寫入記錄檔。

如果您想要對主控台電腦執行資料分析，您需要以系統管理員身分執行 SPA。 當您在本機電腦上執行時，PLA 會經歷不同的程式碼路徑，而這需要系統管理員許可權。

如果您想要針對您的伺服器執行 IIS SPA 建議程式套件，您必須啟用 IIS 伺服器的 WMI 查詢和 ETW 追蹤。 若要這麼做，您可以在 [**健全狀況與診斷**角色服務] 下啟用**追蹤** **，並在**Web 服務器的 [**管理工具**] 底下， ** (iis) **伺服器角色。



### <a name="creating-your-first-project"></a>建立您的第一個專案

設定好所有專案之後，您就可以建立第一個 SPA 專案。 如上一節所述，SPA 會將與效能資料分析相關的所有資訊儲存在資料庫內。 每個 SPA 專案都對應到單一資料庫。 **第一次使用嚮導**會引導您完成下列步驟。

**建立 SPA 專案**

1.  啟動 SpaConsole.exe。 主控台進入中斷連線模式，其中 SPA 未連接到任何資料庫，且主視窗為空白。

2.  若要建立新的專案， **請按一下 [** 檔案]，然後按一下 [ **新增專案**]。 這會啟動第一次使用 Wizard。 第一個頁面會顯示您在使用嚮導時所遵循的步驟：

    * 建立資料庫

    * 布建 advisor 套件

    * 將伺服器新增至目標伺服器清單

3.  按 [下一步] 。 [ **建立專案資料庫** ] 頁面會要求您提供要在其中建立資料庫之 Microsoft SQL Server 實例的名稱。 例如，如果它是在與主控台相同的電腦上，您可以使用**localhost \\ &lt; 的 SQL server 名稱 &gt; **。

    **注意** SQL Server 2008 R2 Express 安裝的預設實例名稱是 SQLExpress。 如果是安裝在本機電腦上 SQL Server 2008 R2 Express 的實例，資料庫通常會預設為 **localhost \\ SQLExpress**。 不過，它可能會在 SQL Server 安裝時發生變更，因此您必須確定使用正確的 SQL Server 實例名稱。



4.  提供資料庫名稱。 只允許字母、數位和底線 (\_) 作為資料庫名稱的有效字元。 Spa 資料庫名稱的合理建議是 **spa**。 如果您輸入不正確名稱，則會出現紅色的錯誤圖示。 相關聯的工具提示會提供驗證失敗的原因。

    **注意** 請務必記住資料庫名稱和伺服器實例名稱，因為這些是您專案的唯一識別碼。 如果您想要切換到這個資料庫，則必須提供此資訊。



5.  提供伺服器實例名稱和資料庫名稱之後，第一次使用 Wizard 會產生資料庫檔案的位置。

6.  在 [ **建立專案資料庫** ] 頁面上，按 **[下一步]**。 第一次使用 Wizard 會建立資料庫，並在資料庫中產生所有 SPA 相關的資料庫架構、函數和預存程式。 視硬體和網路速度而定，此步驟可能需要幾秒鐘的時間。

    **請注意** ，如果此步驟失敗，就會出現錯誤訊息。 常見的問題包括：主控台無法連接到 SQL Server 實例、許可權不足，無法建立資料庫，或資料庫名稱已存在。



7.  當上一個步驟成功時，您會看到 [布建 **advisor 套件** ] 頁面。 它會列出您電腦上所有可用的 advisor 套件。 SPA 會自動掃描 SPA 根目錄下名為「 **ap** 」的資料夾。 它會列出每個 advisor 套件的完整名稱、版本和作者。

    **請注意** ，如需如何在 SPA 中使用完整名稱和版本的詳細資訊，請參閱 [管理 advisor 套件](#bkmk-manageadvisorpacks)



8.  選擇您要在專案資料庫中布建的 advisor 套件，然後按 **[下一步]**。 或者，您可以按一下 [ **略過** ] 移至下一個步驟，而不需要布建任何 advisor 套件。

    **注意** 您隨時都可以在使用此工具時布建 advisor 套件。 如需詳細資訊，請參閱 [管理 advisor 套件](#bkmk-manageadvisorpacks)。



9.  在 [**新增伺服器**] 頁面上，針對要新增至目標伺服器清單的每個伺服器，都有兩個必要欄位要填滿：伺服器和檔案**共用位置****的名稱**。

    **注意** 另外還有一個 **批註** 欄位，主要是用來分類或尋找伺服器。 如果您有多部伺服器，您可以將逗點分隔值 ( .csv) 檔案匯入，其中包含 [伺服器名稱]、[結果] 資料夾和選擇性的 [批註] 欄位。 **批註**欄位是用來描述伺服器，而且可以使用此詞彙來篩選伺服器以進行資料收集。 如果您要透過 .csv 檔案初始化伺服器，檔案內的剖析錯誤不會載入伺服器。

12. 當一切完成時，請按一下 **[完成** ] 以關閉第一次使用 Wizard。 如果您在完成之前關閉第一次使用嚮導，則所有先前的步驟都會保存，而且它們都不會自動復原。 您需要手動進行未來的變更。

## <a name="running-analysis"></a>正在執行分析


設定資料庫之後，您可以在伺服器上執行效能分析。

每次 SPA 主控台啟動時，會自動開啟目前使用者所使用的最後一個專案。 主視窗包含伺服器清單。 每部伺服器都有四個屬性：伺服器名稱、分析結果、目前狀態和批註。

* **伺服器名稱** 伺服器的名稱，也就是伺服器的識別碼。 不允許重複的名稱。

* **分析結果** 根據預設，它會顯示對伺服器執行最新效能分析的結果。 如果未對伺服器執行任何效能分析，則不會顯示任何 **報告**。 如果報表引發警告，則會在產生最新報告時顯示 **警告** 和時間戳記。 如果在伺服器上的最新分析期間找不到任何問題，則會顯示 **[確定]** 和 [時間戳記]。

    **請注意** ，如果您最近變更了系統設定，建議您再次執行分析，以評估變更的整體影響，並取得系統狀態的更新報告。 SPA 不會追蹤受測系統的設定變更。



* **目前狀態** 顯示目前正在伺服器上執行之效能分析工作的狀態。 您可以按一下 [ **取消** ] 圖示（由紅色 X 指定）來取消執行中的工作。

* **批註** 描述目前的目標伺服器。 例如，您可以使用伺服器角色來描述您的伺服器 (例如 SQL Server ) 或位置 (例如 Kent ) 。 SPA 使用 **伺服器名稱** 和 **備註** 來協助搜尋和尋找適當的伺服器。 您可以在 [搜尋] 文字方塊中輸入。 如果 [ **伺服器名稱** ] 或 [ **批註** ] 資料行包含您在 [搜尋] 方塊中輸入的確切字串，伺服器會顯示在 [伺服器] 清單中。

主控台也提供下列控制項：

* **重複** 描述根據時間間隔定期重複集合之功能的核取方塊。 針對大部分的伺服器安裝，您會想要每小時重複一次 SPA 集合，以取得足夠的記錄來進行分析。 如果您只想要執行一次集合，則不應選取 [ **重複** ] 核取方塊。

* **移除週期** 此按鈕可讓您取消進行中的重複收集工作。 它會取消重複的集合，但如果有任何) 正在進行，則不會 (目前的集合。 此選項可讓您重設新的重複收集間隔，或手動執行集合。

開始效能分析之前，請先選取資料收集持續時間。 雖然收集更多資料有助於提供更精確的伺服器效能情況影像，但它也會產生大量的記錄，而且可能會對伺服器造成更多潛在的影響。 根據您的特定需求選擇適當的資料收集持續時間。 每個 advisor 套件都會定義有效的最小持續時間。 您所選擇的資料收集持續時間必須超過所選 advisor 套件的最小持續時間。

若要在目標伺服器上執行效能分析，請選取您要對其執行效能分析的伺服器，然後按一下 [ **執行分析** ] 對話方塊，並要求您選取要在伺服器上執行的 advisor 套件。 選取的 advisor 套件會同時執行。 產生的報表是以相同時間週期內收集的效能資料為基礎。

**注意：** 如果您選取的伺服器有週期性效能分析正在執行，[ **移除週期** ] 按鈕可讓您取消週期性的資料收集。 SPA 不允許在同一部電腦上同時有多個資料收集會話。

### <a name="single-report"></a>單一報表

單一報表會顯示單一電腦上執行 advisor 套件的效能分析結果。 報表會顯示 advisor 套件的名稱、目標伺服器的名稱、報告產生的時間，以及資料收集的持續時間。

單一報表包含通知區段和資料區段。

### <a name="notification-section"></a>通知區段

通知區段是由一組效能分析規則所組成。 每個通知都包含一些資料來源、一些臨界值，以及一些商務邏輯。 當您執行效能分析時，商務邏輯會根據臨界值評估資料來源，以判斷規則是否通過。 如果沒有，則會出現警告，告知您有潛在的效能問題。 它也提供建議來協助您解決問題。 [通知] 區段一律是單一報表檢視中的第一個索引標籤。

通知區段分為兩個部分： **警告** 和 **其他通知**。

如果規則的資料來源符合以邏輯和臨界值設定為基礎的特定條件， **警告** 區域中會出現警告。 警告包含下列部分：

* 警告圖示表示潛在問題是否存在。

* 規則的名稱。 例如， **網路接收封包** 捨棄是指向規則詳細資料頁面的連結，如 [管理 advisor 套件](#bkmk-manageadvisorpacks)中所述。

* 有關潛在問題的簡單描述。

* 可能發生的效能問題解決方案的建議。

不同的伺服器可能會有截然不同的設定和使用模式，而且無法在所有情況下設定適用于所有伺服器的閾值和規則。 SPA 提供修改閾值的功能。 如果規則不適用於您的案例，您也可以選擇停用規則。 預設會啟用所有規則。 停用的規則不會顯示在通知區域中。 如需詳細資訊，請參閱 [管理 advisor 套件](#bkmk-manageadvisorpacks)。

**另一個通知**區域包含所有其他規則，其中不會引發任何警告或規則不適用。 它包含在 **警告** 區域中找到的類似部分。 最大的差異在於，如果未引發任何警告，或規則不適用，則通常不會提供任何建議。

### <a name="data-sections"></a>資料區段

資料區段包含 advisor 套件根據從目標伺服器收集的原始資料所產生的效能資料。 資料區段包含一組最上層區段和多個層級的子區段。 最上層區段會顯示為索引標籤。 最上層區段底下的所有子區段都會顯示在可擴充的區域中。 您可以折迭或展開每個區段，以協助專注于感興趣的區域，如下圖所示。

![資料區段](../media/server-performance-advisor/spa-user-manual-data-sections.png)

核心 OS SPA Advisor 套件和 IIS SPA 建議程式套件包含 **系統總覽** 區段。 本節包含有關資源使用量和設定的最上層資訊。 其他最上層區段代表效能資料的區域。 SPA 以下列方式呈現報表資料：

* **單一值** 成對的索引鍵/值。 索引鍵是字串，代表值的意義。 此值可以是字串、數值或布林值。 這通常用來顯示靜態資訊，例如設定、CPU 架構、記憶體大小總計和 BIOS 版本（不會隨時間而改變）。

* **清單值** 這有時是索引鍵/值組，但清單值可以包含多個欄位。 例如，CPU 的屬性可以顯示在具有多個資料行和多個資料列的資料表中。 每個資料列都代表一個 CPU，而每個資料行都代表 CPU 的屬性。

* **統計資料** 可以被視為一種特殊類型的單一值。 它只能包含數值資料。 在收集資料時，許多數值資料點會變動，而不是保持不變。 例如，每次 PLA 收集效能計數器時，CPU 使用量都會變更。 只顯示單一值無法精確反映效能情況。 這類動態數值資料點不會只顯示一個值、平均、最大值、最小值和90% 的值。 90% 值代表在給定收集間隔內，位於該計數器之所有事件的第90個百分位數的活動。

* **頂端清單** 通常包含特定資源的最上層取用者，或遇到特定事件的最上層實體。 例如，「 **平均 cpu 使用率」的前10個** 程式，包括在資料收集期間，平均 cpu 使用率最高的前十個進程。 因為 CPU 使用量也是動態數值資料點，所以清單中也會包含其他統計資料（例如最大值、最小值和90% 值），以提供使用者更完整的 CPU 耗用量圖片。

如先前章節所述，SPA 會依賴 PLA 來收集 ETW 追蹤、WMI 查詢、效能計數器、登錄機碼和設定檔，以產生報告。 請務必瞭解報表中每個資料點背後的資料來源。 SPA 會透過工具提示提供這類資訊。 您可以將滑鼠停留在索引鍵資料行或資料列上，以查看資料來源工具提示。 例如 **wmi： Win32 \_ DisDrive： Caption** 表示資料來源來自 WMI 查詢、wmi 類別名稱是 Win32 \_ DiskDrive，而屬性為 **Caption**。

### <a name="side-by-side-report"></a><a href="" id="side-by-side-report-"></a>並存報表

單一報表提供通知和資料區段，可協助使用者找出潛在的效能問題，但是通常很難藉由直接查看單一報表來識別潛在的效能問題。 單一報表可能會包含太多資料點，因此難以找出潛在的問題。

為了解決這個問題，SPA 提供了比較兩份報告的功能。 您可以將報表與潛在問題進行比較，以協助找出差異。

您可以從單一報表檢視器啟動並存報表。 使用者可以按一下 [ **動作**]，然後按一下 [ **比較報表** ] 來選取報表。 比較相同的 advisor 套件中的報表，是有意義的。 您可以選擇將報表與先前的報表、時間的下一個報表，或透過搜尋功能選取的任意報表進行比較。 例如，若要隔離異常行為，您可以將基準伺服器報表與在同一部電腦上不同時間產生的報表進行比較，或將報表與在具有類似伺服器角色和載入的不同電腦上產生的報表進行比較。

並列報表看起來非常類似單一報表。 它包含通知區段和資料區段。 它包含與單一報表檢視器相同的通知和資料區段數目。 唯一的差別在於報表會以並列方式顯示。 每個區段都包含來源報表中的資料 (報表 1) 和目的地報表 (報表 2) 。並列報表會顯示 advisor 套件的名稱、目標伺服器的名稱 (報表1的左邊，以及右邊) 的報表2、產生報表的時間，以及每份報表之資料收集的持續時間。

如果您關閉 [ **尋找** ] 對話方塊，您可以鍵入 Control + F 來重新啟用它。此對話方塊會尋找並反白顯示目前區段內的文字字串。

在 [通知] 區段中，如果兩個要比較之報表的任何結果是警告，則會列在 **警告** 區域中。 否則，結果會列在 [ **其他通知** ] 區域中。 因為並存報告的索引鍵是用來識別報表之間的差異，所以不會顯示規則的相關詳細資訊。 使用者可以按一下規則名稱，以顯示規則詳細資料表單，以取得規則的詳細資訊。

在 [資料] 區段中，資料會以並排方式呈現，從左邊的報表1和右邊的報表2資料。 SPA 會在相同的資料表中顯示單一值，但不會加上資料行 **值**的標籤，而是分別命名為 **Report 1** 和 **report 2** 。 並列報表會顯示並存資料表中所有其他形式的資料。

並存報表檢視器也會提供資料來源的相關工具提示。

### <a name="historical-and-trend-charts"></a>歷程記錄和趨勢圖

只有在顯示特定伺服器和特定 advisor 套件的趨勢和歷程記錄圖表時，才有意義。 您必須選擇預設為上周) 的時間範圍 (，然後按一下 **[確定]** ，以顯示 [趨勢] 和 [歷程記錄圖表檢視器]。

趨勢和歷程記錄圖表檢視器有三個索引標籤：歷程記錄圖表、24小時趨勢圖和7天趨勢圖。

### <a name="historical-chart"></a>歷程記錄圖表

歷程記錄圖表會顯示數值資料點在指定時間範圍內的一連串值。 例如，過去15天內單一伺服器上 IIS 的 **平均要求延遲** 。 歷程記錄圖表中的每個資料點都代表一個效能分析會話中所建立的特定資料來源值。

有幾種方式可使用歷程記錄圖表：

1.  為了協助您在特定時間針對資料點（例如，每天上午2:00）尋找異常狀況，IIS 的 **平均要求延遲** 會從200毫秒到500毫秒。

2.  以協助將多個資料點相互關聯。 例如，顯示過去15天內 **平均要求延遲** 和 **平均要求計數** 。 報表可能會顯示要求延遲和要求計數會在相同的時間點增加，這可能表示要求延遲增加是因為要求計數增加所造成。

在歷程記錄圖表中，使用者可以執行下列動作：

* 在圖表區域中顯示多個資料數列。 每個資料數列在報表檢視器中會顯示為折線圖。 每個折線圖都會自動調整以納入報表檢視器中。

* 從歷程記錄圖表檢視器底部的資料序列清單中，加入或移除資料數列。

* 顯示或隱藏資料數列清單中的資料數列。 使用者可以按一下清單中的特定資料數列，以反白顯示圖表區域中對應的折線圖。

* 選取圖表區域內的時間週期，以放大特定的時間週期。 若要縮小，請按一下位於圖表左下角的按鈕。

* 按兩下特定資料點，以調查單一報表。

* 複製資料並讓它可供其他程式使用，例如 Microsoft Excel。 這可讓您在適當的情況下使用 Microsoft Excel 圖表製作功能。

### <a name="trend-charts"></a>趨勢圖

許多重複的效能問題是因為在或針對目標伺服器執行的定期工作所造成。 例如，以娛樂為導向的網站可能會在週末期間獲得更多的點擊，或排程磁片備份工作可能會在每天上午2:00 關閉伺服器的效能。

趨勢圖的設計目的是要協助您找出這類效能問題。 效能問題可能會在各種模式中重複發生。 最常見的模式是每日模式和每週模式，也就是在同一小時內或一周相同的一天發生效能問題。 因此，SPA 提供24小時的趨勢圖和7天的趨勢圖。

趨勢分析圖形針對一組資料提供更深入的調查，並根據一天中的時間尋找趨勢。 X 軸是設定為24小時制，從0:00 開始 (午夜) ，結束于23:59。 SPA 不會同時顯示多個資料數列的趨勢。 您可以按一下 [ **挑選資料序列** ]，選取要查看的資料數列。

為了處理資料，SPA 會尋找每小時在0:00 到0:59 之間取得的所有快照集。 SPA 會針對該小時內所建立的快照集，判斷其最小值、最大值、平均值和 sigma 值，並將其繪製為 k 線圖。 SPA 會針對在1:00 到1:59 之間建立的快照集重複執行程式，然後再對2:59 進行2:00，依此類推。 如果指定的小時內沒有任何快照集，SPA 會將該小時保留在圖形上，然後繼續進行下一個小時。

7天的趨勢圖非常類似24小時趨勢圖。 唯一的差別在於它會根據一周中的某一天而不是一天的小時來分組資料數列。

您在趨勢和歷程記錄圖表中選取的資料數列會儲存為使用者喜好設定。 下一次為相同的 advisor 套件開啟趨勢和歷程記錄圖表檢視器時，會將相同的一組資料數列列為預設值。

## <a name="managing-reports"></a>管理報表


### <a name="deleting-reports"></a>刪除報表

您可以移除報表，以將需要由 SPA 管理的報表數目降至最低。 根據報告的頻率和伺服器數目，建議您刪除不必要的報表。 雖然 SPA 沒有可管理之報表的限制，但基礎資料庫可能具有大小限制。

**注意：** 刪除的報表無法刪除。



### <a name="exporting-and-importing-reports"></a>匯出和匯入報表

您可以將報表匯出至 XML 檔案，以傳輸到另一個 SPA 主控台或傳送電子郵件給另一位使用者。 匯出報表並不會刪除報表。 若要匯出目前已查看的報表，請在 **報表檢視器**中按一下 [ **動作**]，然後按一下 [ **匯出**]。 若要匯出多個報表，請從 **報表瀏覽器**按一下 [ **啟用多重選取**]，從選取方塊中選取多個報表，然後按一下 [ **匯出**]。 這會將 XML 格式的報表匯出至選取的目的地目錄。

匯出的報表可以在 SPA 中查看。 匯入的報表不會加入 SPA 資料庫中。 它們主要是作為匯出報表的 XML 檢視器應用程式。 匯入之報表的伺服器不需要安裝相同的 advisor 套件作為原始匯出的報表 SPA 主控台。

## <a name="managing-advisor-packs"></a><a href="" id="bkmk-manageadvisorpacks"></a>管理 advisor 套件


SPA 包含適用于核心作業系統、Hyper-v、active directory 和 IIS 的 advisor 套件。 SPA 提供使用 SQL 開發 advisor 套件的開放架構，因此非 Microsoft 開發人員也可以建立 advisor 套件的版本。 有四個選項可管理 advisor 套件：布建、自訂、重設或移除。

### <a name="provision-new-advisor-packs"></a>布建新的 advisor 套件

Microsoft 或非 Microsoft 開發人員可以發行新的 advisor 套件。 Advisor 套件包含 provisionMetaData.xml 和一組描述邏輯的 SQL 腳本。

**布建新的 advisor 套件**

1.  複製 *% SpaRoot%* ap 目錄下的 advisor 套件的所有內容 \\ 。

2.  在主視窗中 **，按一下 [** 設定]，然後按一下 [ **設定 Advisor 套件**]。 [ **設定 Advisor 套件** ] 對話方塊隨即開啟。

    **注意** 此對話方塊類似于第一次使用 Wizard 中的 [布建 **advisor 套件** ] 頁面。 它會顯示可用來管理的 advisor 套件清單。 清單中的每個 advisor 套件都有屬性，例如名稱、已安裝的版本、版本和作者。 名稱是 advisor 套件的完整名稱，而安裝的版本是已在專案中布建的此 advisor 套件版本。 如果未在目前的資料庫中布建 advisor 套件，則 **會**顯示 [已安裝的版本] 文字方塊。 [版本] 欄位會指出此 advisor 套件的版本，其會在 [advisor 套件] 資料夾下進行。



3.  從清單中選取 advisor 套件。 如果尚未布建 advisor 套件，或 advisor 套件資料夾中有較新的版本，則會啟用 [布建 **] 按鈕。** 按一下 [ **布** 建] 按鈕。

4.  布建完成時，所選 advisor 套件的 [ **已安裝的版本** ] 欄位就會包含新的版本資訊。

### <a name="customize-advisor-packs"></a>自訂 advisor 套件

SPA 定義了開放的架構，可讓使用者修改 advisor 套件。 使用者可以藉由變更臨界值、共用閾值，以及啟用或停用規則來修改 advisor 套件檔案。

如需如何變更和建立 advisor 套件的詳細資訊，請參閱 [伺服器效能建議程式套件開發指南](server-performance-advisor-pack-development-guide.md)。

### <a name="changing-thresholds"></a>變更臨界值

在 SPA 中使用閾值來判斷是否符合規則的觸發條件。 實際客戶案例的實際臨界值可能會因為工作負載、硬體環境和商務需求而有極大的差異。 預設臨界值對目前的使用者案例而言可能不正確，所以 SPA 提供修改現有閾值的功能。

您可以在單一或並存報表中，按一下規則名稱來修改臨界值。 或者，若要管理特定 advisor 套件的所有規則 **，則可以從 [設定** ] 功能表修改臨界值。

**變更臨界值**

1.  在 [設定] 功能表中， **按一下 [** **設定 advisor 套件**]，按一下要修改的 Advisor 套件名稱，然後按一下 [ **設定**]。

    **注意** 您會看到 advisor 套件中包含的所有規則清單。 Advisor 套件名稱左邊的核取方塊會指出是否已啟用規則。 如果規則已停用，則會隱藏在所有報表中。



2.  按一下您想要修改的特定規則。 所選規則的 [ **規則詳細資料** ] 表單隨即開啟。

[規則詳細資料] 表單包含特定規則的詳細資訊。 其中包括名稱、描述、狀態、可能的結果和臨界值。 SPA 支援兩種類型的規則結果： **警告** 和 **確定**。 每種類型都有建議的文字和建議。

某些規則未定義臨界值。 例如， **HTTP 保持** 運作規則會檢查 IIS 的布林值設定。 因此臨界值清單可能是空的。 否則，會列出目前規則所使用的所有臨界值。 有關如何在規則中使用閾值的詳細說明，包含在描述中。

如果從 [設定 **] 功能表啟動 [** **規則詳細資料**] 表單，則閾值清單會有三個數據行： [名稱]、[原始設定] 和 [變更] 設定。 如果是從單一或並存報告啟動，則也會包含報表所使用的臨界值。 使用者可以藉由變更 [ **變更設定** ] 資料行中的值，然後按一下 [ **儲存** ]，將變更儲存至資料庫，以修改目前的臨界值。

對臨界值所做的所有變更只會套用至變更後所產生的報告。 這些變更不會影響現有的報表。

### <a name="sharing-thresholds"></a>共用閾值

如果您在類似的情況下管理伺服器，您可以選擇使用一組相同的臨界值。 您可以 **使用 [設定** ] 功能表來匯出和匯入特定 advisor 套件的閾值。 您可以選取特定的 advisor 套件，然後按一下 [ **設定**]。 匯出的閾值檔案為 XML 格式。

匯入閾值時，SPA 會驗證 XML 檔案格式，並驗證檔案是否符合選取的 advisor 套件。 如果成功，SPA 會將臨界值檔案中的所有值匯入到目前的專案資料庫中。 與先前的變更臨界值案例類似，所有閾值的變更只會在未來產生的報表上生效。 現有的報表不會受到影響。

### <a name="enable-or-disable-rules"></a>啟用或停用規則

您可以從 [ **規則詳細資料** ] 表單啟用或停用規則。 您需要按一下 [ **儲存** ] 來保存所做的變更。 如果規則已停用，則不會顯示在任何報告中。 但是基礎商務邏輯會在產生報告時觸發，因此當您選擇重新啟用規則時，就會再次顯示在報告中。

### <a name="reset-advisor-packs-to-original-state"></a>將 advisor 套件重設為原始狀態

您可能會決定修改資料庫中已布建的 advisor 套件。 除了變更臨界值以外，您也可以變更 SQL 腳本。 SPA 不支援變更報表中繼資料，或新增或移除布建的 advisor 套件的規則。 手動修改這些區域可能會導致非預期的行為。

如需變更布建的 advisor 套件的詳細資訊，請參閱 [伺服器效能建議程式套件開發指南](server-performance-advisor-pack-development-guide.md)。

如果您想要復原在已布建的 advisor 套件上所做的變更，您可以選擇重設 advisor 套件。 這會覆寫與 advisor 套件相關的所有 SQL 腳本，並重設所有的預設臨界值。 這會保留所有現有的報表。

您可以使用 [ **設定 Advisor 套件** ] 表單來重設 advisor 套件。 您必須選取要重設的 advisor 套件，然後按一下 [ **重設**]。

### <a name="remove-advisor-packs"></a>移除 advisor 套件

當不再需要 advisor 套件時，使用者可以從資料庫中將它移除。 移除 advisor 套件會從資料庫中移除 advisor 套件的所有相關資訊，包括規則和臨界值、所有 SQL 腳本，以及所有報表。 無法回復任何動作。

您可以使用 [ **設定 Advisor 套件** ] 表單來移除 advisor 套件。 您必須選取要移除的 advisor 套件， **然後按一下 [** 取消布建]。

### <a name="update-existing-advisor-packs"></a>更新現有的 advisor 套件

更新現有的 advisor 套件非常類似于將 advisor 套件重設為其原始狀態。 若要將 advisor 套件更新為較新的版本，請將新的 advisor 套件複製到 advisor 套件資料夾。 只有在沒有報表中繼資料變更時，advisor 套件才會被視為現有 advisor 套件的更新。 報表和規則識別碼必須相同。 否則，應該將它們視為兩個不同的 advisor 套件。

如果只有商務邏輯變更，而且沒有任何建議程式套件的報表中繼資料變更，則應該提供新的版本號碼，例如從1.0 到2.0。 如果有報表中繼資料變更，建議程式套件應提供不同的完整名稱。 例如，您可以將 ServerPerformanceAdvisor 變更為 ServerPerformanceAdvisor （可能為 IIS. V2）。

如果有較新版本的 advisor 套件，[設定建議程式 **套件** ] 表單中的清單會自動以最新版本的 advisor 套件填滿 **版本** 資料行。 您可以選取 advisor 套件，然後按一下 [ **重設**]。 Advisor 套件會以新的商務邏輯和臨界值進行更新。 此 advisor 套件的所有報表都會保留下來。

### <a name="automating-spa-report-collection-by-using-windows-powershell"></a>使用 Windows PowerShell 自動化 SPA 報表收集

下列程式表示如何使用 SPA Windows PowerShell Cmdlet 將 SPA 報表集合自動化的範例。

使用者認證可透過 Windows PowerShell 進行加密和快取。 這些認證會用來登入遠端伺服器。 雖然並非 SPA 專用，但下列範例會示範這項技術：

``` syntax
$fileName = 'D:\temp\operator.txt'

```

### <a name="use-t-sql-to-generate-reports"></a>使用 T-sql 來產生報表

您可以使用 SQL 來解壓縮 SPA 報表資料，以建立不是在 SPAConsole.exe 內提供的自訂報表。

例如，下列 T-sql 命令會針對涵蓋過去三天的時間範圍，提供依 CPU 排列的前10部伺服器清單：

``` syntax
select TOP 10
   MachineName,
   AverageCpu
FROM (
   select
      __MachineName MachineName,
      AVG(AverageValue) AverageCpu
   FROM [SPA].[Microsoft.ServerPerformanceAdvisor.CoreOS.V1].vwCpuPerformance
   WHERE __Creationtime > dateadd(DAY, -3, GETUTcdate())
      AND Name = N'% Processor time' AND CpuId = N'_Total'
   GROUP BY __MachineName
) t
OrdER BY t.AverageCpu DESC
```

### <a name="working-with-multiple-projects"></a>使用多個專案

在某些情況下，您可能會想要分割 SPA 所使用的 SQL Server 資料庫。 這可協助減少 SQL Server 資料庫的資料大小，或協助您以邏輯方式分割伺服器。 在這些情況下，SPA 主控台支援多個專案。 您可以按一下 [檔案 **]，然後按一下 [****新增專案**]，以建立新的專案資料庫。 第一次使用 Wizard 會出現，以協助建立新的專案資料庫。

建立專案之後，您可以按一下 [檔案]，然後按一下 [**開啟專案** **]，在**專案之間切換。 當目前開啟的專案內執行未完成的效能分析時，SPA 主控台不允許切換資料庫。 這是為了保護效能資料的完整性。

當您選擇建立新的專案資料庫或開啟不同的專案資料庫時，將會關閉目前的專案。 當目前專案關閉時，會關閉所有開啟的報表。

**注意** SQL Server 2008 R2 Express 的資料庫限制為 10 GB。 藉由使用多個專案，您可以使用一或多個 SQL Server 資料庫，並維持在 10 GB 的 SQL Server 2008 R2 Express 限制下。



### <a name="logging-and-debugging"></a>記錄和調試

SPA 提供基本的記錄功能。 它只允許將記錄寫入記錄檔，該檔案位於與 SPAConsole.exe 相同的資料夾中。 執行 SPA 主控台的使用者帳戶必須具有安裝 SPA 之資料夾的寫入權限，以確保記錄可以寫入記錄檔。

SPA 包含下列有效的記錄層級：

* **參考資訊** 傾印 SPA 主控台所採用的每個動作的記錄，而且主要是針對偵錯工具用途而設計。

* **警告** 記錄 SPA 主控台內發生的所有失敗和例外狀況。 某些失敗只是可由 SPA 主控台處理的驗證失敗。

* **重大** 只記錄 SPA 主控台無法處理的失敗和例外狀況。 這些失敗會導致 SPA 主控台損毀。 記錄檔會提供這類失敗的內容資訊。

根據預設，記錄層級是警告，這表示 SPA 只會記錄 SPA 中發生的失敗和例外狀況。 您可以藉由在 SpaConsole.exe 所在的相同資料夾中編輯 **SpaConsole.exe.config** 檔案，來變更記錄層級。 所有記錄檔都會寫入至相同資料夾中的 log.txt 檔案。

SPA 也提供一些基本功能來進行調試。 若要開啟 SPA 的偵錯工具，使用者需要手動修改 SPA 專案資料庫。 設定會儲存在設定資料表中。 使用者必須執行下列 SQL 腳本，才能將 SPA 專案變更為「偵測模式」：

``` syntax
UPdate [Configurations] SET Value = N'true' WHERE Name = N'Debugmode'.
```

在「偵測模式」中，SPA 架構會保留在效能分析期間產生的所有暫存資料，讓您可以針對 advisor 套件中的問題進行疑難排解。 此腳本主要是設計供 advisor 套件開發人員使用。

如需 SPA 中偵錯工具功能的詳細資訊，請參閱 [Server Performance Advisor 套件開發指南](server-performance-advisor-pack-development-guide.md)。

SPA 主控台不提供備份和還原 SPA 資料庫的功能。 您應使用標準 Microsoft SQL Server 工具和腳本來備份和還原資料庫。

## <a name="privacy-and-security"></a>隱私權和安全性


SPA 僅支援從目標伺服器到 SPA 主控台的資料收集。 它不是為了將任何資訊傳送給 Microsoft 或非 Microsoft 開發人員而設計的。 如需 SPA 隱私權的詳細資訊，請參閱 Server Performance Advisor 的 Microsoft 軟體授權條款。

SPA 所收集的所有資料都會儲存在 project 資料庫中。 由於某些 ETW 追蹤的本質，SPA 可能會收集可能具有高商業重要性的機密資訊。 您應留意與共享 SPA 專案資料庫存取相關聯的潛在風險。 暫存記錄檔會儲存在每部目的電腦上指定的共用資料夾下。 即使在資料匯入完成時，SPA 會嘗試刪除這些暫存記錄檔，但並不保證一定會刪除這些記錄檔。 您應確認共用資料夾位於安全的位置。

SPA advisor 套件包含 SQL 腳本，可剖析及分析效能記錄以產生效能報告。 SPA 會嘗試限制這些腳本執行的許可權。 不過，仍有可能腳本可以從目標伺服器透過 SPA 收集機密資訊，或取得或修改儲存在相同 SPA 專案資料庫中的機密資訊。 您必須確定布建至 SPA 專案資料庫的所有 advisor 套件都是來自信任的來源。

## <a name="errors-and-troubleshooting"></a>錯誤和疑難排解


### <a name="spaconsoleexe-does-not-start-or-write-log-file"></a>SPAConsole.exe 不會啟動或寫入記錄檔

當您第一次嘗試執行 SPAConsole.exe 時，如果未安裝 .NET Framework，應用程式就不會啟動或寫入記錄檔。 在您啟動 SPA 之前，請確定已安裝 compatible.NET Framework 並正常運作。

### <a name="locating-log-information"></a>尋找記錄資訊

SPA 會將失敗資訊儲存在 SPA 資料夾下的 log.txt 檔案中。 詳細的錯誤訊息和呼叫堆疊資訊會寫入此資料夾。 如果您遇到需要更多資訊來解讀的失敗，可以開啟 log.txt 檔案以查看錯誤詳細資料。

### <a name="database-size-limitations-for-sql-server-express"></a>SQL Server Express 的資料庫大小限制

SQL Server Express 的大小限制為 10 GB，代表使用者資料庫。 此大小資料庫具有儲存 20000-30000 報告的容量。 若要降低達到此資料庫限制的可能性，建議您定期刪除報表，並/或使用另一個沒有 10 GB 使用者資料庫限制的 SQL Server 版本。

### <a name="sql-server-express-log-size-and-disk-capacity"></a>SQL Server Express 記錄檔大小和磁片容量

如果您使用 SQL Server Express，則使用者資料庫限制為 10 GB，但對應的記錄檔可能會超過 70 GB。 基於這些理由，我們建議 SQL Server Express 100 GB 以上的可用磁碟空間。 此磁碟空間應該足以儲存大約20000到30000的報告。 此記錄檔的名稱為 SPADB \_ ，在 **% Program Files% \\ Microsoft SQL Server \\ mssql10.mssqlserver 找到。SQLEXPRESS \\ MSSQL \\ 資料**。

### <a name="failure-to-connect-to-target-server"></a>無法連接到目標伺服器

當您在目標伺服器上執行效能分析時，執行 SPA 主控台的使用者帳戶必須具有特定許可權，才能將資料收集器集合人員排入目標伺服器上的 PLA。 您也必須變更 Windows 防火牆設定，才能讓 PLA 通訊通過。 如需詳細的設定指示，請參閱 [設定 SPA](#bkmk-setupspa)。

### <a name="failure-to-create-pla-data-collection-set-on-the-target-server"></a>無法在目標伺服器上建立 PLA 資料收集組

如果您無法在目標伺服器訊息上建立 PLA 資料收集組，請務必執行下列動作：

* 確定 **效能記錄 & 警示** 服務正在執行

* 安全性設定 [ **網路存取：不允許儲存密碼和認證以進行網路驗證** ] 已停用。 必須停用安全性設定，因為 SPA 需要使用使用者認證，才能在目標伺服器上建立資料收集組。

### <a name="running-spa-against-the-console"></a><a href="" id="running-spa-against-the-console-"></a>針對主控台執行 SPA

如果目標伺服器與 SPA 主控台相同，則 PLA 會經歷不同的通道。 即使使用者帳戶正在以系統管理員許可權執行 SPA 主控台，但 PLA 卻失敗。 如果 SPA 主控台已安裝在目標伺服器上，使用者需要以系統管理員身分啟動 SPA，以確保效能分析工作可以在主控台上執行。

### <a name="running-multiple-consoles-at-the-same-time"></a>同時執行多個主控台

SPA 不支援同時針對相同 SPA 專案資料庫執行的多個主控台。 SPA 也不提供鎖定和同步處理機制來防止發生此情況。 如果有兩個 SPA 主控台同時執行，則根據這些 SPA 主控台執行的時間序列，主控台會以不一致的方式運作。 為了避免這種情況，所有已排入佇列的效能分析會話都應該先從清單中移除，然後才由啟動分析的主控台進行處理。

SPA 可保護 SPA 成功產生的每個報告的完整性。 同時，SPA 不保證所有已排入佇列的分析工作都已完成。 如果您看到效能分析會話的狀態變更不一致，或是宣告系統找不到資料收集器集合人員所產生的效能記錄，可能是因為針對相同 SPA 專案資料庫執行的多個 SPA 主控台實例所造成。

執行 SPA Windows PowerShell Cmdlet 也可能受針對相同 SPA 資料庫執行的 SPA 主控台所影響。 建議您先關閉 SPA 主控台，再執行 SPA Windows PowerShell Cmdlet。

### <a name="spaconsoleexe-recurring-collection-is-disrupted"></a>SPAConsole.exe 週期性收集中斷

當您執行 SPAConsole.exe 並使用週期性的資料收集時 (例如) 的每小時集合，執行 SPAConsole.exe 的伺服器不應處於省電模式，使其可以暫停。 SPA 不會檢查是否有省電原則。 暫止的活動可能會中斷定期資料收集。

### <a name="lost-etw-events"></a>遺失的 ETW 事件

若要在目標伺服器上建立最小效能影響，請將 PLA 設計成在收集效能資訊時以低優先順序執行。 如果目標伺服器忙碌中，則 PLA 可以捨棄一些資料收集工作，以產生在目標伺服器上執行的高優先順序工作。 您應該考慮設定共用資料夾，其中事件是在不與工作負載 s i/o 或更快的磁片磁碟機（例如 SSD）發生衝突的磁片上寫入。 當事件被捨棄時，報表會在報表檢視中報告，因為遺失的事件可能會影響所產生效能度量的可靠性。

某些許可權問題也可能會略過登錄或 WMI 查詢。 不過，這比 ETW 事件遺失更不可能發生。 如此一來，資料收集器集合的結果有時不會包含所有要求的值。 您必須確保所有 advisor 套件的 T-sql 腳本都會處理此情況。 如果資料不存在於資料收集結果中，它就會在報表中標示為沒有資料。

由於 ETW 事件遺失對 PLA 來說很常見，因此根據 ETW 追蹤產生的資料點可能不會與根據所產生的資料點一致，例如，效能計數器。 例如，您可以看到 IIS 的總 CPU 使用量是 80% (這是來自) 的效能計數器，而頂端的 Url 只會使用 CPU 時間的 10% (這是來自 ETW 追蹤) 的資料點。 通常，一個資料點的資料來源可以透過資料點的工具提示來查看。 您應該留意這類資料遺失的影響。

若要避免這類事件遺失，應關閉目標伺服器的結果資料夾。

如果資料收集器的結果包含 ETW 追蹤遺失以外的不完整資料，而 advisor 套件開發人員已新增 ETW 事件遺失通知的支援，則會在單一報告上顯示資訊列，以通知使用者可能因為資料遺失而產生的不一致報表。 您可以在 log.txt 檔案中找到詳細的資料遺失資訊。

## <a name="glossary"></a>詞彙


以下是搭配 SPA 使用的一些詞彙：

* **Advisor 套件** 中繼資料和 T-sql 腳本的集合，可處理從目標伺服器收集的效能記錄。 Advisor 套件接著會從效能記錄資料產生報告。 Advisor 套件中的中繼資料會定義要從目標伺服器收集以進行效能測量的資料。 中繼資料也會定義一組規則、臨界值和報表格式。 最常見的情況是，建議程式套件專門針對單一伺服器角色撰寫，例如 Internet Information Services (IIS) 。

* **Spa 主控台** SpaConsole.exe，這是 spa 的核心部分。 SPA 不需要在您要測試的目標伺服器上執行。 SPA 主控台包含 SPA 的所有使用者介面，從設定專案到執行分析和查看報表都是如此。 SPA 在設計上是兩層式應用程式。 SPA 主控台包含 UI 層和一部分的商務邏輯層。 SPA 主控台會排定並處理效能分析要求。

* **SPA 架構** 提供所有使用者介面、效能記錄檔處理、設定、錯誤處理和資料庫 Api，以及管理程式。

* **SPA 專案** 此資料庫包含在 advisor 套件的目標伺服器上產生之目標伺服器、advisor 套件和效能分析報表的所有相關資訊。 您可以比較和查看相同 SPA 專案內的歷程記錄和趨勢圖。 您可以建立一個以上的專案。 SPA 專案彼此獨立，不會在專案之間共用任何資料。

* **目標伺服器** 使用特定伺服器角色（例如 IIS）執行 Windows Server 的實體電腦或虛擬機器。

* **資料分析會話** 在特定目標伺服器上進行效能分析。 資料分析會話可以包含多個 advisor 套件。 來自這些 advisor 套件的資料收集器集合程式會合並成單一資料收集器集合。 單一資料分析會話的所有效能記錄都會在相同的時間週期內收集。 分析在相同資料分析會話中執行的 advisor 套件所產生的報表，可協助使用者瞭解整體效能狀況，並找出效能問題的根本原因。

* **Windows 事件追蹤** Windows 中提供的高效能、低負擔、可擴充的追蹤系統。 它會提供程式碼剖析和偵錯工具，可用於針對各種案例進行疑難排解。 SPA 使用 ETW 事件做為產生效能報告的資料來源。 如需 ETW 的一般資訊，請參閱 [使用 Etw 改善偵錯工具和效能微調](/archive/msdn-magazine/2007/april/event-tracing-improve-debugging-and-performance-tuning-with-etw)。

* **Windows Management Instrumentation (WMI) ** Windows 中管理資料和作業的基礎結構。 您可以撰寫 WMI 腳本或應用程式，將遠端電腦上的系統管理工作自動化。 WMI 也會提供管理資料給作業系統的其他部分和產品。 SPA 使用 WMI 類別資訊和資料點做為產生效能報表的來源。

* **效能計數器** 用來提供作業系統或應用程式、服務或驅動程式如何執行的相關資訊。 效能計數器資料有助於判斷系統瓶頸並微調系統和應用程式效能。 作業系統、網路和裝置提供應用程式可取用的計數器資料，讓使用者能夠以圖形方式查看系統的執行狀況。 SPA 使用效能計數器資訊和資料點做為來源，以產生效能報告。

* **效能記錄檔及警示 (PLA) ** 收集效能記錄和追蹤，並在符合特定觸發程式時引發效能警示。 PLA 可以用來收集效能計數器、Windows 的事件追蹤 (ETW) 、WMI 查詢、登錄機碼和設定檔案。 PLA 也支援透過遠端程序呼叫， (RPC) 進行遠端資料收集。 使用者定義資料收集器集合程式，其中包括要收集之資料的相關資訊、資料收集的頻率、資料收集持續時間、篩選，以及儲存結果檔案的位置。 SPA 使用 PLA 收集目標伺服器的所有效能資料。

* **單一報表** 根據單一目標伺服器上一個 advisor 套件的一個資料分析會話所產生的 SPA 報告。 它可以包含通知和各種資料區段。

* **並存報表** 一種 SPA 報表，可比較相同 advisor 套件的兩個單一報表。 這兩份報告可以從不同的目標伺服器產生，或從相同目標伺服器上的個別效能分析執行。 並存報表會建立比較兩份報告的功能，以協助使用者識別其中一個報表中的異常行為或設定。 並列報表包含通知和各種資料區段。 在每個區段中，這兩個報表的資料會並存列出。

* **趨勢圖** SPA 報表，用來調查效能問題的重複模式。 許多重複的效能問題是因為排程的伺服器從伺服器或用戶端電腦載入變更所造成，而這可能會在每日或每週發生。 SPA 提供24小時的趨勢圖，以及7天的趨勢圖來識別這些問題。

    使用者可以一次選擇一或多個資料數列，也就是單一報表內的數值，例如 **平均 CPU 使用率總計**。 更明確地說，數值是單一伺服器的純量值，該伺服器是由指定時間實例的單一 AP 所產生。 SPA 會將這些值分組到24個群組中，一天的每個小時會有一個 (七天的報表，每一周的每一天) 。 SPA 會計算每個群組的平均、最小值、最大值及標準差。

* 歷程**記錄圖表**SPA 報表，可用來顯示給定伺服器和 advisor 套件組在一段時間內特定數值的變更。 使用者可以選擇多個資料數列，並在歷程記錄圖表中一起顯示，以瞭解不同資料數列之間的相互關聯。

* **資料數列** 在一段時間內從相同資料來源收集的數值資料。 相同的來源表示資料必須來自相同的目標伺服器，例如在一部伺服器上的 IIS 平均要求佇列長度。

* **規則** 邏輯、閾值和描述的組合。 它們代表潛在的效能問題。 每個 advisor 套件包含多個規則。 每個規則都是由報表產生進程所觸發。 規則會將邏輯和臨界值套用至單一報表中的資料。 如果符合條件，則會引發警告通知。 如果沒有，則會將通知設定為 [ **確定]** 狀態。 如果規則不適用，則會將通知設定為不適用的 (**NA**) 狀態。

* **通知** 規則顯示給使用者的資訊。 它包含規則的狀態 (**[確定]**、[ **NA**] 或 [ **警告**) ]、規則的名稱，以及解決效能問題的可能建議。
