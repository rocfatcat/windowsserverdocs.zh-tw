# Windows Admin Center UI 文字與設計樣式指南

>適用於： Windows Admin Center

本主題說明撰寫 Windows Admin Center 使用者介面 (UI) 文字的一般方法，以及一些我們所採用的特定慣例和方法。

Windows Admin Center 及任何擴充功能都應該遵循 [Microsoft 的語氣表達原則](https://docs.microsoft.com/style-guide/brand-voice-above-all-simple-human)，讓人在體驗上感覺易於使用且友善。 本樣式指南以下列語氣表達原則為依據，同時也參酌 [Microsoft 書寫樣式指南](https://docs.microsoft.com/style-guide/welcome/)來擬定，因此請務必查看這兩項資源以取得[協助工具](https://docs.microsoft.com/style-guide/accessibility/accessibility-guidelines-requirements)、[縮略字](https://docs.microsoft.com/style-guide/acronyms)和[用詞](https://docs.microsoft.com/style-guide/word-choice/) (例如[敬請](https://docs.microsoft.com/style-guide/a-z-word-list-term-collections/p/please)和[抱歉](https://docs.microsoft.com/style-guide/a-z-word-list-term-collections/s/sorry)) 等要項的相關資訊。

## 按鈕

- 按鈕應盡可能只用一個字詞，特別是您要將工具當地語系化的話。 兩個或三個是正常，但是嘗試避免再。 如果您有四個字或較長時間，它會是較好的使用連結控制項。
- 按鈕標籤應簡潔、明確且不言而喻。 與其泛用一般的「送出」按鈕，不如使用對應於使用者動作的動詞，例如「建立」、「刪除」、「新增」、「格式化」等。
- 如果按鈕承接前面的問題，其標籤應清楚對應該問題 (通常為「是」或「否」)。

## 大寫

我們遵循 Microsoft 樣式的[大寫](https://docs.microsoft.com/style-guide/capitalization)：對於所有內容幾乎都使用句子樣式大寫。

| UI 元素              |大寫|註解|
|-------------------------|--------------|--------|
|徽章 (例如 PREVIEW) |全部大寫      ||
|所有其他項目          |句子樣式|但我們顯露 WMI 或 PowerShell 的物件屬性所在位置會有幾處例外。|

## 冒號

使用冒號引入清單。 例如：

    Choose one of the following:
    Cats
    Dogs
    Quokkas

不要使用冒號 UI 文字標籤在開啟時，不同的行件事從它的標籤或它時清楚的區別標籤和件事加上標籤。

標籤是在同一行做為文字，它標籤，而您需要將兩個元素一起執行時，請使用 UI 文字中的冒號。

## 確認訊息

確認對話方塊就非常實用，繼續進行可能會有非預期的結果，例如資料遺失。 它們應該包含容易掃視、 有用的資訊，以清除結果，特別是針對無法還原的事件。 

- 請務必確認是必要的。 如果不沒有提供任何新資訊 （例如，「 確定嗎？ 」） 則可能不需要確認訊息。  
- 確認客戶想要繼續執行該動作。
- 請確定主要指示 （標題），並說明文字 （本文） 不是重複。
- 在標題，定義可能的結果為問題或接下來會發生的相關的陳述式。 例如，「 清除此磁碟機上的所有資料？或者 「 您即將清除您的所有資料 」。
- 在本文中新增詳細資料。 如果有都是變數，例如您已變更，相關之項目的名稱包含在此。
- 包含一個簡單的問題 （無論是在標頭，或在本文中） 的畫面清除在兩個動作按鈕之間選擇。
- 對於複雜的選項，請使用是/否鼓勵仔細閱讀的按鈕。 對於較簡單的選項，使用按鈕的特定動作，例如全部刪除，或取消。

## 初次執行體驗

使用者第一次來到頁面時，您有機會協助他們開始使用您的工具。 這可能是：

- 空白頁面中的文字字串，簡短說明如何開始使用，例如「選取 [新增] 以新增應用程式」。
- 控制項的連結，讓使用者開始著手操作，例如「新增應用程式來開始使用」。
- 一段告訴使用者如何開始使用的小巧簡短動畫或影片

以下是節錄自我們 Windows 樣式指南的一些提示：

### 1. 切合實用

- 避免行銷作風和語言。
- 當您示範或建議一些動作時，請確保最終結果明確。如果只是告知客戶做動作的方式，他們卻不知道這麼做的原因時，不會有效果。
- 如果客戶不需要提示，就別提供。

### 2. 顯示但不指示

文字盡可能保持簡單 (試想一下小動畫或視訊)。

### 3. 不要過於頻繁

- 將快顯與提示次數限制為每個使用工作階段總計 4 次，包括系統通知和殼層通知。
- 確定快顯出現時正好用得上。
- 不要妨礙客戶正常操作。
- 確定快顯可輕鬆關閉。

### 4. 保持內容相關

- 教學最具果效的時間就是那些適時提供指引的片刻。
- 如果您建立教學課程或投影片，要保持資訊具體而實際。
- 不要行銷的「夸夸其辭」，只要專注在特定提示與訣竅。
- 為客戶提供稍後返回教學課程的方式，如果涉及相關情況時 (使用者第一次通常不會保留資訊，但安裝指示可能也只涉及一次)。
- 空白狀態訊息是很自然讓使用者學習和/或開心的地方，請保持其簡單且具參考性。

### 5. 盡量減少費事的設定程序

當您需要客戶再多執行一個動作來體驗完整價值 (登入線上服務等等) 時，盡可能使之輕鬆愉快。

- 訊息應簡短且直接。
- 避免把他們送走。 儘可能提供可讓他們隨處連線過來的方式。
- 如果可以，允許使用者選擇稍後做動作，晚些時候再提醒他們這麼做。
- 如果您帶他們離開其原先體驗，請提供能快速且輕鬆切換回去的方式。

## 說明連結

以下是節錄自我們 Windows 樣式指南的一些提示：

### 何時我們應該提供的說明連結？

幾乎永遠不會。 提供的說明連結只能在：

- 沒有客戶很可能有的回應將會協助他們在 UI 工作成功在 UI 中使用時明顯也是重要問題。 
- 以提供資訊讓使用者能在將 UI 工作成功所需的量，在 UI 中沒有足夠的空間。 

### 其中應該可以幫助連結出現？ 

- 文字連結應該會出現，那麼靠近說明儘可能針對 UI 元素。 
- 如果您必須提供文字連結可套用至整個 UI 螢幕，請將它放在畫面的左下方。 
- 如果您提供的連結，透過 [說明] 按鈕 （？），在工具提示應該是 「 說明 」。

### 我們應該使用哪些 URL？

永遠不會直接連結至 web 位址 — 改為使用重新導向服務。

Microsoft 開發人員應該使用 FWLink 除了當它是使用者可能需要手動輸入，在哪些案例使用 aka.ms/project-rome 連結 （只要 URL 的目標是瀏覽器地區設定，例如 Docs.microsoft.com 會自動辨識的網站） 的說明連結

### 文字的指導方針 

- 使用完整的句子。
- 不包含結尾標點符號除了問號。 
- 您不需要使用相同的文字，做為工作標題;使用 UI 的內容中有意義的文字，但請確定是兩者之間的邏輯連線。 例如： 
- 說明連結︰ 允許的例外狀況的風險是什麼？ 
- 說明主題標題: 」 允許透過 Windows 防火牆與通訊程式 」
- 請盡有關說明主題的內容。 
    - 我們的樣式
        - Windows 防火牆如何協助保護我的電腦？
        - 為什麼醒目提示可以改善一張圖片
    - 不是我們的樣式
        - Windows 防火牆的相關詳細資訊
        - 深入了解色彩管理
        - 深入了解
- 使用完整的句子做為連結的文字，而不只是金鑰字詞。 
    - 我們的樣式 
        - [允許例外狀況的風險有哪些？]()
    - 不是我們的樣式
        - [允許例外狀況的風險]()是什麼？ 

## 錯誤訊息

以下是從 Windows 樣式指南，適應一些指導方針：

撰寫良好的訊息是提供足夠的說明，但不是過度技術; 之間取得平衡之間於隨性和 personable 但不是擾人或具有冒犯意味。

### 一般指導方針

使用錯誤案例每一則訊息。

#### 標題

- 讓它保持簡潔有力並說明簡明扼要問題是，或**項目理想的情況下執行**。 <br>某些 UI 表面可能會有標題，而不是包裝在太長時截斷，請特別留意對於這類的。
- 如果它是一個簡單的步驟，請在標題中使用的解決方案。
- 請確定以防讀取器會忽略內文文字直接到按鈕與關聯的標題。
- 避免使用 「 發生問題 」 中的標題，除非您會不有任何其他選擇。 可以更特定的相關問題。
- 避免使用變數 （例如檔案、 資料夾和應用程式名稱） 中的標題。 將它們放在本文中。

#### Body

- 如果標題夠說明的問題或方案，您不需要內文文字。
- 不要重複些微差異訊息中的標題。
- 清楚地通訊，並簡明扼要的解決方案是什麼。
- 焦點放在第一次進行的事項。
- 錯誤運作不要怪罪使用者。
- 如果有錯誤程式碼與錯誤相關聯，如果您認為可能有助於包括錯誤代碼，客戶或 Microsoft 支援服務以研究問題，包含它內文文字的正下方，並將它寫入，如下所示：

    錯誤碼: # # #

    如果客戶有解決錯誤沒有程式碼所需的所有資訊，您不需要包含它。

#### 按鈕

- 撰寫按鈕的文字，這樣才能是主要指令的特定回應。 如果不可行，使用 [關閉] 關閉按鈕文字 （而不是 「 無妨 」 或 「 執行 」）。
- 如果您有多個按鈕，讓最左邊的按鈕鼓勵使用者採取的動作。 請最右側按鈕更保守的動作，例如 「 [取消]。 」

#### 說明連結

僅限，請考慮您無法讓特定和可採取動作的錯誤訊息的說明連結。

## Null 狀態文字

以下是一些說明從 Windows 樣式指南。

客戶資料或內容不存在時，會發生 null 狀態從應用程式或功能，搜尋，或必要時，會傳回任何結果資訊時遺失表單，例如帳單交易的資訊。

### 指導方針

 - 如果可能的話，將有機會作為 null 狀態的情況下教育使用者有關如何使用功能 （例如，如何新增音樂，其中尋找圖片等等。）  
- 如果您在 UI 中有標題，將說明 「 修正 」 （例如，「 新增一些音樂 」） 的 null 狀態採取的動作 
- 有有趣的文字。 這個空間可以提供愉悅經驗，因為它可能不會看到數次。 
- 避免 「 它是孤獨在這裡。 」 這是悲傷，而且有已過度使用。 
- 避免發生問題，例如 「 還沒有連接印表機 」？ 確定要使用一次，但通常會取得過度使用此格式和問題放在客戶的額外負荷/壓力。 它也可以感覺 condescending。 
- 各種不同 null 狀態文字中的是個好消息。 

### 範例

- 「 將其他人新增為我的最愛，及您會看到以下 」。
- [了解任何成就或您特別的感到自豪遊戲短片嗎？ 將它們新增到您的展示。 」
- 「 沒有人的廠商尚未中。 啟動一個 ！ 」
- 「 當人做為朋友加入您時，您會看到它們這裡。 」
- 「 當您執行更像是解除鎖定成就、 錄製遊戲短片，並新增朋友時，您會看到它這裡。 」
- 「 這裡，將會顯示您最愛的朋友，因此您可以看到它們是線上什麼它們是最多和。 」

## 標點符號

- 標題或不完整的句子沒有結束標點符號 (句號、問號)。 在標題有提問的確認對話方塊中，則屬例外
- 在[句號](https://docs.microsoft.com/style-guide/punctuation/periods)和[問號](https://docs.microsoft.com/style-guide/punctuation/question-marks)上使用 Microsoft 樣式指南的指引。

## 狀態訊息

狀態訊息包含快顯 (快顯通知) 訊息與通知。

|字串類型         | 備註                               |
|------------        |-------------------------------------|
|快顯通知               |有結束標點符號的句子案例：最好使用物件變數，讓使用者可以在他們已巡覽離開物件時，了解訊息套用於什麼物件|
|通知標題|沒有結束標點符號的句子案例 (即標題)：最好使用物件變數|
|通知詳細資料|完整句子，最好有顯示物件之 UI 的連結|

以下是一些對通知訊息的詳細建議：

|字串類型         | 備註                               |
|------------        |-------------------------------------|
|已開始             |盡可能省略：通常可以跳至「進行中」訊息，將分心的干擾次數降到最少。|
|進行中         |以正在執行之動作的動詞開頭，並在結尾使用省略符號表示持續進行的作業。 以下是範例：<br> *正在建立磁碟區 "客戶資料"...*|
|成功             |以「已成功」開頭，並以軟體剛才完成的作業為結尾。 以下是範例：<br> *已成功建立磁碟區 "客戶資料"。*|
|失敗             |以「無法」開頭，並以為軟體無法完成的作業為結尾。 以下是範例：<br> *無法建立磁碟區 "客戶資料"。*|

## 工具提示

良好的工具提示的簡短描述無標籤的控制項或有標記的控制項，提供一些額外的資訊時，這會有用。 他們也可以協助客戶瀏覽由提供額外的 UI — 不重複 — 控制項標籤、 圖示、 連結、 等等的相關資訊。

謹慎或完全不應該使用工具提示。 它們可以是客戶，因此不包含在工具提示，只需重複標籤或狀態明顯中斷。 它應該一律新增寶貴的資訊。

|    內容                                 |    如何撰寫工具提示    |
|    -----------------------                 |    -------------------------    |
|當控制項或 UI 元素是未標記...|使用簡單、 描述性的名詞片語。 例如：<br> 反白顯示 [手寫筆 |
|當標示為 UI 元素，但它的用途需要釐清...|<ul><li>簡短描述您可以使用此 UI 元素中執行的動作。 </li><li>使用命令式動詞表單。 例如，「 文字中找到這個檔案 」 （不 」 文字中尋找此檔案 」）。</li><li>請勿包含結尾標點符號，除非有多個完整的句子。</li> </ul>|
|當文字標籤會被截斷，或者可能會在某些語言截斷...|<ul><li>提供的工具提示中的 untruncated 的標籤。</li><li>選用： 在另一個行，釐描述，但只是如果需要提供。</li><li>沒有提供工具提示，如果 untruncated 的資訊頁面或流程上提供其他地方。</li></ul>|
|如果使用鍵盤快速鍵...|<ul><li>選用： 提供在括號內下列標籤或描述性的片語，例如 「 列印 (Ctrl + P) 」 或 「 尋找此檔案 (Ctrl + F) 中的文字 」 的鍵盤快速鍵</li><li>它是 [確定] 以新增有用的鍵盤快速鍵來釐工具提示，但請避免將新增工具提示僅顯示鍵盤快速鍵。 </li></ul>|
