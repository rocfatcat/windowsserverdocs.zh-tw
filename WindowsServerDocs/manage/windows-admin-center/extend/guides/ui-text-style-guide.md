# <a name="windows-admin-center-ui-text-and-design-style-guide"></a>Windows Admin Center UI 文字與設計樣式指南

>適用於：Windows Admin Center

本主題說明撰寫 Windows Admin Center 使用者介面 (UI) 文字的一般方法，以及一些我們所採用的特定慣例和方法。

Windows Admin Center 及任何擴充功能都應該遵循 [Microsoft 的語氣表達原則](https://docs.microsoft.com/style-guide/brand-voice-above-all-simple-human)，讓人在體驗上感覺易於使用且友善。 本樣式指南以下列語氣表達原則為依據，同時也參酌 [Microsoft 書寫樣式指南](https://docs.microsoft.com/style-guide/welcome/)來擬定，因此請務必查看這兩項資源以取得[協助工具](https://docs.microsoft.com/style-guide/accessibility/accessibility-guidelines-requirements)、[縮略字](https://docs.microsoft.com/style-guide/acronyms)和[用詞](https://docs.microsoft.com/style-guide/word-choice/) (例如[敬請](https://docs.microsoft.com/style-guide/a-z-word-list-term-collections/p/please)和[抱歉](https://docs.microsoft.com/style-guide/a-z-word-list-term-collections/s/sorry)) 等要項的相關資訊。

## <a name="buttons"></a>按鈕

- 按鈕應盡可能只用一個字詞，特別是您要將工具當地語系化的話。 有二或三個沒問題，但請試著避免較長的時間。 如果您有四個單字或更長的字，最好使用連結控制項。
- 按鈕標籤應簡潔、明確且不言而喻。 與其泛用一般的「送出」按鈕，不如使用對應於使用者動作的動詞，例如「建立」、「刪除」、「新增」、「格式化」等。
- 如果按鈕承接前面的問題，其標籤應清楚對應該問題 (通常為「是」或「否」)。

## <a name="capitalization"></a>大寫

我們遵循 Microsoft 樣式的[大寫](https://docs.microsoft.com/style-guide/capitalization)：對於所有內容幾乎都使用句子樣式大寫。

| UI 元素              |大寫|註解|
|-------------------------|--------------|--------|
|徽章 (例如 PREVIEW) |全部大寫      ||
|所有其他項目          |句子樣式|但我們顯露 WMI 或 PowerShell 的物件屬性所在位置會有幾處例外。|

## <a name="colons"></a>冒號

使用冒號來引進清單。 例如:

    Choose one of the following:
    Cats
    Dogs
    Quokkas

當標籤位於標記所在的另一行，或標籤與其標記的內容有清楚的區別時，請勿在 UI 文字中使用冒號。

當標籤位於與標記文字相同的行上，而且您需要讓兩個元素一起執行時，請在 UI 文字中使用冒號。

## <a name="confirmation-messages"></a>確認訊息

當繼續執行時可能會有非預期的結果（例如資料遺失）時，確認對話方塊會很有用。 他們應該包含具有清楚結果的可流覽、有用的資訊，特別是針對無法反轉的事件。 

- 請確定確認是必要的。 如果沒有要提供的新資訊（例如「您確定嗎？」），則可能不需要確認訊息。  
- 確認客戶想要繼續執行此動作。
- 請確定主要指示（標題）和解說文字（主體）不是多餘的。
- 在標題中，將可能的結果定義為問題或語句，以瞭解接下來會發生什麼事。 例如，「清除此磁片磁碟機上的所有資料嗎？」或「您即將清除所有資料」。
- 在主體中新增詳細資料。 如果有變數，例如您要變更的專案名稱，請將它包含在此處。
- 包含一個簡單的問題（在標頭或主體中），這兩個動作按鈕之間會有清楚的選擇。
- 針對複雜的選擇，請使用 [是]/[否] 按鈕，以鼓勵仔細閱讀。 如需更簡單的選擇，請使用動作特有的按鈕，例如 [全部刪除] 或 [取消]。

## <a name="first-run-experiences"></a>首次執行體驗

使用者第一次來到頁面時，您有機會協助他們開始使用您的工具。 這可能是：

- 空白頁面中的文字字串，簡短說明如何開始使用，例如「選取 [新增] 以新增應用程式」。
- 控制項的連結，讓使用者開始著手操作，例如「新增應用程式來開始使用」。
- 一段告訴使用者如何開始使用的小巧簡短動畫或影片

以下是節錄自我們 Windows 樣式指南的一些提示：

### <a name="1-be-helpful"></a>1.有幫助

- 避免行銷作風和語言。
- 當您示範或建議某個專案時，請確定最終結果是清除的;只有當客戶不知道其執行方式時，才會顯示如何執行某項作業的方法。
- 如果客戶不需要提示，請不要提供秘訣。

### <a name="2-show-dont-tell"></a>2.顯示，不要分辨

文字盡可能保持簡單 (試想一下小動畫或視訊)。

### <a name="3-dont-overwhelm"></a>3.不嚴重

- 將快顯與提示次數限制為每個使用工作階段總計 4 次，包括系統通知和殼層通知。
- 確定快顯出現時正好用得上。
- 不要讓客戶執行某些動作。
- 確定快顯可輕鬆關閉。

### <a name="4-keep-it-contextual"></a>4.保持內容的關聯

- 教學最具果效的時間就是那些適時提供指引的片刻。
- 如果您建立教學課程或投影片，要保持資訊具體而實際。
- 不要行銷的「夸夸其辭」，只要專注在特定提示與訣竅。
- 提供一種方法，讓客戶稍後返回教學課程（如果相關的話）（人們通常不會在第一次保留資訊，但設定指示可能只會與此相關）。
- 空白狀態訊息是很自然讓使用者學習和/或開心的地方，請保持其簡單且具參考性。

### <a name="5-minimize-painful-setup"></a>5.將痛苦設定降至最低

當您需要客戶再多執行一個動作來體驗完整價值 (登入線上服務等等) 時，盡可能使之輕鬆愉快。

- 訊息應簡短且直接。
- 避免把他們送走。 儘可能提供可讓他們隨處連線過來的方式。
- 如果可以，允許使用者選擇稍後做動作，晚些時候再提醒他們這麼做。
- 如果您帶他們離開其原先體驗，請提供能快速且輕鬆切換回去的方式。

## <a name="help-links"></a>說明連結

以下是節錄自我們 Windows 樣式指南的一些提示：

### <a name="when-should-we-provide-a-help-link"></a>何時應該提供說明連結？

幾乎不會。 只有在下列情況時，才提供說明連結：

- 客戶可能會在使用者介面中有很明顯的問題，也就是可協助他們在 UI 工作成功的答案。 
- UI 中沒有足夠的空間可提供使用者在 UI 工作成功完成所需的資訊數量。 

### <a name="where-should-help-links-appear"></a>哪裡應該會出現說明連結？ 

- 文字連結應該會出現在 UI 元素附近，說明會盡可能地導向。 
- 如果您必須提供適用于整個 UI 畫面的文字連結，請將它放在畫面的左下方。 
- 如果您透過 [說明] 按鈕（？）提供連結，則工具提示應該是 [說明]。

### <a name="what-url-should-we-use"></a>我們應該使用哪個 URL？

永遠不要直接連結到網址，改為使用重新導向服務。

Microsoft 開發人員應該使用 FWLink，除非是使用者可能必須手動輸入的說明連結，在這種情況下，請使用 aka.ms 連結（只要 URL 的目標是自動辨識瀏覽器地區設定的網站，例如 Docs.microsoft.com）

### <a name="text-guidelines"></a>文字指導方針 

- 使用完整句子。
- 除了問號以外，請勿包含結束標點符號。 
- 您不需要使用與工作標題相同的文字;使用在 UI 內容中有意義的文字，但請確定兩者之間有邏輯連接。 例如: 
- 說明連結：允許例外狀況的風險為何？ 
- 說明主題標題：「允許程式透過 Windows 防火牆進行通訊」
- 與說明主題的內容盡可能具體。 
    - 我們的風格
        - Windows 防火牆如何協助保護我的電腦？
        - 重點為何可以改善圖片
    - 不是我們的風格
        - Windows 防火牆的詳細資訊
        - 深入瞭解色彩管理
        - 深入了解
- 將整個句子用於連結文字，而不只是關鍵字。 
    - 我們的風格 
        - [允許例外狀況的風險為何？]()
    - 不是我們的風格
        - [允許例外]()狀況的風險為何？ 

## <a name="error-messages"></a>錯誤訊息

以下是從 Windows 風格指南中調整的一些指導方針：

撰寫良好的訊息是提供足夠的說明，但不太技術性的平衡;在偶爾和 personable 之間，但不是討厭或冒犯。

### <a name="general-guidelines"></a>一般指導方針

針對每個錯誤案例使用一則訊息。

#### <a name="headings"></a>放

- 請將它保持簡短，並清楚說明問題所在或最**理想**的作法。 <br>有些 UI 表面可能會有截斷而不是換行的標題，因此請留意這些。
- 如果這是一個簡單的步驟，請在標題中使用解決方案。
- 請確定標題會直接與按鈕相關，以防讀者忽略本文文字。
- 除非您沒有其他選擇，否則請避免在標題中使用「發生問題」。 更明確地瞭解此問題。
- 避免在標題中使用變數（例如檔案、資料夾和應用程式名稱）。 將它們放在主體中。

#### <a name="body"></a>本文

- 如果標題足以說明問題或解決方案，您就不需要本文文字。
- 不要以稍微不同的用語重複訊息中的標題。
- 以清楚且簡潔的方法進行溝通。
- 請先專注于提供事實。
- 不要讓使用者的錯誤發生。
- 如果有與錯誤相關聯的錯誤碼，而且您認為包含錯誤碼可能有助於客戶或 Microsoft 支援人員調查問題，請將其包含在本文文字正下方，並如下所示撰寫：

    錯誤碼：####

    如果客戶具有在沒有程式碼的情況下解決錯誤所需的所有資訊，您就不需要包含它。

#### <a name="buttons"></a>按鈕

- 寫入按鈕文字，使其成為主要指令的特定回應。 如果無法這麼做，請使用「關閉」作為關閉按鈕文字（而不是「良好」或「完成」）。
- 如果您有一個以上的按鈕，請將最左邊的按鈕設為建議使用者採取的動作。 將最右邊的按鈕設為較保守的動作，例如「取消」。

#### <a name="help-links"></a>說明連結

請只考慮您無法明確且可採取動作之錯誤訊息的說明連結。

## <a name="null-state-text"></a>Null 狀態文字

以下是 Windows 樣式指南的一些協助。

當應用程式或功能中沒有客戶資料或內容、搜尋後未傳回任何結果，或表單缺少必要資訊（例如交易的帳單資訊）時，就會發生 Null 狀態。

### <a name="guidelines"></a>指導方針

- 可能的話，請使用 null 狀態的情況，讓人們學習如何使用此功能（例如，如何新增音樂、尋找圖片的位置等等）。  
  - 如果您的 UI 中有標題，請說明「修正」 null 狀態所採取的動作（例如「新增一些音樂」） 
  - 讓文字更有趣。 這個空間可能是提供取悅的機會，因為它可能不會出現數次。 
  - 請避免「它在這裡是孤立的」。 這很悲傷，而且已過度用。 
  - 避免「尚未連接您的印表機？」等問題 可以使用一次，但這種格式通常會取得濫用性，而問題則會對客戶造成額外的負擔/壓力。 它也可以感受到 condescending。 
  - 各種不同的 null 狀態文字是件好事。 

### <a name="examples"></a>範例

- 「將某人新增為我的最愛」，您會在這裡看到。」
- 「有任何成就或遊戲剪輯您特別自豪嗎？ 將它們新增至您的展示。」
- 「一方還沒有人。 啟動一個 up！ "
- 「當有人將您加入成為朋友時，您會在這裡看到。」
- 「當您執行像是解除鎖定成就、錄製遊戲剪輯及新增朋友等東西時，您會在這裡看到一切。」
- 「您最愛的朋友會在此顯示，因此您可以看到他們在線上的時間，以及他們的最新資訊。」

## <a name="punctuation"></a>標點符號

- 標題或不完整的句子沒有結束標點符號 (句號、問號)。 在標題有提問的確認對話方塊中，則屬例外
- 在[句號](https://docs.microsoft.com/style-guide/punctuation/periods)和[問號](https://docs.microsoft.com/style-guide/punctuation/question-marks)上使用 Microsoft 樣式指南的指引。

## <a name="status-messages"></a>狀態訊息

狀態訊息包含快顯 (快顯通知) 訊息與通知。

|字串類型         | 注意                               |
|------------        |-------------------------------------|
|快顯通知               |有結束標點符號的句子案例：最好使用物件變數，讓使用者可以在他們已巡覽離開物件時，了解訊息套用於什麼物件|
|通知標題|沒有結束標點符號的句子案例 (即標題)：最好使用物件變數|
|通知詳細資料|完整句子，最好有顯示物件之 UI 的連結|

以下是一些對通知訊息的詳細建議：

|字串類型         | 注意                               |
|------------        |-------------------------------------|
|已啟動             |盡可能省略：通常可以跳至「進行中」訊息，將分心的干擾次數降到最少。|
|進行中         |以正在執行之動作的動詞開頭，並在結尾使用省略符號表示持續進行的作業。 以下為範例：<br> *正在建立磁片區「客戶資料」 。*|
|成功             |以「已成功」開頭，並以軟體剛才完成的作業為結尾。 以下為範例：<br> *已成功建立「客戶資料」磁片區。*|
|失敗             |以「無法」開頭，並以為軟體無法完成的作業為結尾。 以下為範例：<br> *無法建立磁片區「客戶資料」。*|

## <a name="tooltips"></a>工具提示

良好的工具提示會簡短描述未標記的控制項，或提供一些額外的資訊給已標記的控制項，這是有用的。 他們也可以藉由提供其他（而非多餘）有關控制項標籤、圖示、連結等的資訊，協助客戶流覽 UI。

應該謹慎使用工具提示，或根本不需要。 它們可能會中斷客戶，因此請勿包含只會重複標籤或陳述明顯的工具提示。 它應該一律會加入寶貴的資訊。

|    內容                                 |    如何撰寫工具提示    |
|    -----------------------                 |    -------------------------    |
|當控制項或 UI 元素未標記時 。|使用簡單的描述性名詞片語。 例如:<br> 反白顯示畫筆 |
|當 UI 元素加上標籤，但其目的需要澄清 。|<ul><li>簡要描述您可以使用此 UI 元素執行的動作。 </li><li>使用命令式動詞形式。 例如，「尋找此檔案中的文字」（不是「在這個檔案中尋找文字」）。</li><li>除非有多個完整句子，否則請勿包含結束標點符號。</li> </ul>|
|當文字標籤被截斷或可能在某些語言中截斷時 。|<ul><li>在工具提示中提供 [untruncated] 標籤。</li><li>選擇性：在另一行中，請提供說明的描述，但只有在必要時才需要。</li><li>如果頁面或流程的其他位置提供 untruncated 資訊，請勿提供工具提示。</li></ul>|
|如果可以使用鍵盤快速鍵 。|<ul><li>選擇性：在標籤或描述片語後面的括弧中提供鍵盤快速鍵，例如「列印（Ctrl + P）」或「尋找此檔案中的文字（Ctrl + F）」</li><li>您可以將有用的鍵盤快速鍵加入至 [說明] 工具提示，但請避免只新增工具提示來顯示鍵盤快速鍵。 </li></ul>|
